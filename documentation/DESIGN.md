# 博渊 OA — 业务规则与权限设计

> **文档职责**：定义建筑工程版的组织架构、角色权限、审批流、薪资计算规则和数据保留策略。这是当前唯一预置方案（建筑工程版）的完整业务设计，同时也是 `preset.sql` 种子数据的业务依据。凡文中标注"可配置"的项，均可由指定角色在系统运营期修改。
>
> **目标读者**：产品经理、后端开发者、前端开发者。
>
> **关联文档**：实体字段定义和 API 路由见 `platform/ARCHITECTURE.md`；代码实现见 `platform/BACKEND_IMPL.md`；页面布局见 `presets/construction/UI_DESIGN.md`（业务页面）和 `platform/UI_DESIGN.md`（平台Shell）。

## 1. 组织架构与人员体系

### 1.1 概念模型

系统中的人员由四个**独立正交**的维度描述：

```
员工 Employee（系统的最小主体单元）
 ├── 岗位 Position      你是什么人、干什么活、怎么算薪资、有哪些功能开关
 ├── 等级 Level         你在这个岗位里处于哪个薪资段（可选）
 ├── 角色 Role          你在系统里能做哪些操作（权限控制）
 └── 直系领导 Supervisor 你向谁汇报、谁审批你的单据
```

| 混淆场景 | 正确理解 |
|---------|---------|
| "项目经理"既是岗位也是系统权限 | 岗位="项目经理"控制薪资/功能；角色=`project_manager` 控制审批权限，二者通常对齐但可以分离 |
| "劳工"是特殊员工类型 | 劳工只是 `requiresConstructionLog=true` 的岗位，系统里所有人统一为"员工" |
| 薪资规则应该配到角色上 | 薪资规则配到岗位/等级，角色只控制操作权限 |

---

### 1.2 岗位配置（Position）

岗位是薪资规则、功能开关、假期配置的主要载体。字段定义见 `ARCHITECTURE.md`，以下为建筑工程版预置岗位：

| positionCode | positionName | employeeCategory | defaultRoleCode | requiresConstructionLog | hasPerformanceBonus |
|-------------|-------------|-----------------|----------------|------------------------|---------------------|
| `CEO` | CEO | OFFICE | `ceo` | false | true |
| `FINANCE` | 财务 | OFFICE | `finance` | false | true |
| `PM` | 项目经理 | OFFICE | `project_manager` | false | true |
| `OFFICE_STAFF` | 办公人员 | OFFICE | `employee` | false | true |
| `WORKER` | 劳工 | LABOR | `worker` | true | false |
| `WELDER` | 焊工 | LABOR | `worker` | true | false |
| `PLUMBER` | 管工 | LABOR | `worker` | true | false |

**等级示例（焊工）：**
```
岗位：焊工（baseSalary=6000）
  等级 L1（初级）：baseSalary=5500，annualLeaveDays=5
  等级 L2（中级）：baseSalary=7000，annualLeaveDays=7
  等级 L3（高级）：baseSalary=9000，annualLeaveDays=10
```

---

### 1.3 配置继承链（薪资规则取值优先级）

薪资规则、社保配置、假期配置等所有可配置项遵循以下优先级（高优先级覆盖低优先级）：

```
优先级 1（最低）：全局默认值
    ↓ 可覆盖
优先级 2：员工类别层（OFFICE / LABOR 各自的类别默认值）
    ↓ 可覆盖
优先级 3：岗位层（Position Config）
    ↓ 可覆盖
优先级 4：等级层（Level Config，等级存在时生效）
    ↓ 可覆盖
优先级 5（最高）：个人覆盖（Employee 个人配置，需 CEO 审批）
```

| 层级 | 可修改人 | 是否需要审批 | 生效时机 |
|-----|---------|------------|---------|
| 全局默认值 | CEO | 无需审批 | 立即 |
| 员工类别层 | CEO | 无需审批 | 立即 |
| 岗位配置 | CEO / 财务 | 财务修改需 CEO 审批 | 下个结算周期 |
| 等级配置 | CEO / 财务 | 同上 | 下个结算周期 |
| 个人覆盖 | CEO / 财务 | 财务修改需 CEO 审批 | 下个结算周期 |

---

### 1.4 组织树路由规则

每个员工有唯一直系领导，构成树形汇报关系，根节点为 CEO。

**直系领导取值优先级（由高到低）：**
1. 员工档案中显式指定的 `directSupervisorId`
2. 员工岗位配置的 `supervisorPositionCode` → 取该岗位的在职负责人
3. 所属项目的项目经理（若员工有项目归属）
4. CEO（兜底，所有路径失败时）

**组织树用途：**

| 用途 | 说明 |
|-----|------|
| 审批路由 | 请假/加班找直系领导初审，无 PM 时自动上浮到 CEO |
| 数据范围 | 项目经理可查看本项目员工数据 |
| 通知推送 | 加班通知、结算提醒沿组织树推送 |
| 汇报展示 | 经营总览的人员组织图 |

**无直系领导时的兜底规则：**

| 场景 | 节点1（初审）处理方式 |
|-----|---------------------|
| 员工无直系领导且无归属项目 | 自动转给 CEO |
| 员工归属项目但项目无 PM | 自动转给 CEO |

> CEO 执行了节点1（初审）后，节点2（终审）仍由 CEO 执行（CEO 可连续审批同一单据的两个节点，属于兜底设计）。

**初始化组织架构示例：**
```
CEO
 └── 财务（汇报给 CEO）
 └── 项目经理 A（汇报给 CEO）
      └── 现场工长（汇报给项目经理 A）
           └── 焊工 × N（汇报给现场工长）
 └── 项目经理 B
      └── ...
```

## 2. 角色定义

### 2.1 内置角色（不可删除，不可修改 roleCode）

| roleCode | 显示名称 | 定位 |
|---------|---------|------|
| `ceo` | CEO | 企业最高管理者，全局数据访问，所有业务终审，系统配置 |
| `finance` | 财务 | 薪资结算执行方，人员档案协管，财务数据维护 |
| `project_manager` | 项目经理 | 项目范围内初审，施工日志审批与模板管理 |
| `employee` | 员工 | 普通办公室员工，发起个人申请，查看本人数据 |
| `worker` | 劳工 | 现场施工人员，额外有施工日志和工伤补偿提交权 |

**自定义角色规则：**
- CEO 可新增自定义角色，从内置权限集中勾选
- 每名员工当前版本只分配**一个主角色**
- 高风险权限（终审审批、解锁结算更正、配置审批流、配置数据保留策略）不可分配给自定义角色
- 自定义角色可删除（删除前需先解除人员绑定）

### 2.2 数据范围

| 角色 | 数据范围 | 说明 |
|-----|---------|------|
| `employee` | SELF | 只能操作本人申请、档案、工资条 |
| `worker` | SELF | 同上；施工日志和工伤补偿限定于本人所属项目 |
| `project_manager` | PROJECT | 只作用于本人负责的项目及该项目下的员工 |
| `finance` | GLOBAL | 财务相关数据全局可见；不含施工日志等非财务数据 |
| `ceo` | GLOBAL | 全局所有数据 |

## 4. 权限边界说明

### 4.1 无直系领导时的审批兜底

所有审批流程（请假、加班、工伤）节点1均由直系领导初审。当申请人无法确定直系领导时，节点1自动转给 CEO。CEO 执行节点1后，节点2（终审）仍由 CEO 执行（连续两节点，系统不拦截，属兜底设计）。

---

### 4.2 工伤补偿的初审路由规则

工伤补偿节点1审批人依据**提交人身份**动态决定：

| 提交人 | 节点1 初审人 | 说明 |
|-------|------------|------|
| worker 本人 | project_manager（同项目） | 标准路径 |
| project_manager（代录） | **跳过节点1**，直接进入 CEO 终审 | 提交人即初审人，skipCondition 触发 |
| finance（代录） | project_manager（同被提交人项目） | 财务不参与现场核实，仍由 PM 初审 |
| 其他员工/劳工（代录） | project_manager（同被提交人项目） | 路由同 finance 代录 |
| 无项目经理时（任意情形） | CEO（兜底） | — |

---

### 4.3 财务编辑角色权限的限制

`finance` 可编辑角色权限，但系统**不允许**为任何角色开启以下高风险权限：

- 终审审批
- 审批结算更正（解锁周期）
- 配置审批流
- 配置数据保留策略
- 新增/删除自定义角色

以上权限项在财务的角色管理页中显示为只读。

---

### 4.4 加班通知制的例外路径

所有员工不可自主申报加班，统一由 PM 或 CEO 发起通知。**例外路径**：当 PM 漏发通知时，员工可提交自补加班申请（附证明材料），走 PM → CEO 双审批。

| 场景 | 说明 |
|-----|------|
| PM 发起通知 → CEO 审批 | 标准路径，需 CEO 终审 |
| CEO 直接发起通知 | 自动归档，无需额外审批 |
| 员工自补申请（PM 漏报） | 需附证明材料，走 PM 初审 + CEO 终审 |
| 无归属项目的员工自补 | PM 节点自动转 CEO，由 CEO 单审 |

员工可在通知中标记"未参与"并说明原因，由 PM 最终确认是否接受拒绝。

---

### 4.5 施工日志提交前提

`worker` 角色员工提交施工日志时，必须已被分配到至少一个**进行中的项目**。若无项目归属，提交按钮灰显，提示"请联系管理员分配项目"。

---

### 4.6 运营期配置修改的审批约束

非 CEO 角色对以下配置的修改均需提交 CEO 审批，**下一个结算周期生效**：

| 配置类型 | 可发起修改的角色 | 生效时机 |
|---------|----------------|---------|
| 岗位配置（薪资/假期/社保规则） | finance | 下个结算周期 |
| 等级配置 | finance | 下个结算周期 |
| 员工个人薪资覆盖值 | finance | 下个结算周期 |
| 绩效奖金个人覆盖值 | finance | 下个结算周期 |
| 全局薪资规则（倍率/比例） | finance | 下个结算周期 |

CEO 直接修改以上任何配置均无需审批；社保模式切换需在系统中确认生效时机提示。

## 5. 审批流配置

### 5.1 请假申请（LEAVE）

**流程：**
```
[员工提交]
    ↓ PENDING
[节点1: 直系领导 初审，项目/部门范围；无直系领导时转 CEO]
    ↓ APPROVING
[节点2: CEO 终审，全局]
    ↓
APPROVED → ARCHIVED（自动纳入薪资结算扣款计算）
REJECTED → 申请人可查看驳回原因，重新提交（新单据，历史保留）
```

**配置参数：**

| 参数 | 值 |
|------|---|
| `businessType` | `LEAVE` |
| 节点数 | 2 |
| 节点1 | 角色: `directSupervisor` \| 范围: `PROJECT_OR_DEPT` |
| 节点2 | 角色: `ceo` \| 范围: `GLOBAL` |
| 追溯申报 | 任意时刻可补申报；审批通过后扣款计入**审批完成当月**工资单 |

---

### 5.2 加班申请（OVERTIME）

> **加班采用通知制**：所有员工不可自主申报加班，统一由 PM 或 CEO 发起。

**PM 发起路径：**
```
[PM 发起加班通知（选项目/成员/日期/说明）]
    → 通知发送给选中员工，员工默认同意；可拒绝并说明原因，由 PM 终审
    ↓ PENDING → [CEO 终审] → APPROVED → ARCHIVED（计入薪资结算）
```

**CEO 直接发起：**
```
[CEO 发起加班通知] → 自动 APPROVED → ARCHIVED
```

**员工自补路径（PM 漏报时）：**
```
[员工补报，需上传证明材料] → PENDING → [PM 初审] → [CEO 终审] → APPROVED → ARCHIVED
```

> 加班跨月结算：审批完成后计入**审批完成当月**工资单，与请假追溯规则一致。

**配置参数：**

| 参数 | 值 |
|------|---|
| `businessType` | `OVERTIME` |
| 发起方 | `project_manager` \| `ceo`（员工自补路径除外） |
| CEO 发起 | 自动归档，无终审节点 |

---

### 5.3 工伤补偿（INJURY）

**代录规则**：任何员工可代他人提交工伤补偿申请，填写被提交人工号即可。

**初审路由**：见 §4.2。

**流程：**
```
[提交] → PENDING
[节点1: PM 初审（依路由规则，PM代录时跳过）]
    ↓ APPROVING
[节点2: CEO 终审]
    ↓
APPROVED → ARCHIVED（记录工伤事实，进入待理赔状态）
    → 财务在保险理赔完成后录入理赔金额，关联至目标薪资周期（可跨月）
REJECTED → 申请人可修改后重新提交
```

> 理赔金额录入不在申请表单中，由财务在工伤审批归档后、保险理赔结束时录入。

**配置参数：**

| 参数 | 值 |
|------|---|
| `businessType` | `INJURY` |
| 节点1跳过条件 | 提交人 role == `project_manager`（skipCondition: `SUBMITTER_ROLE_MATCH`） |
| 节点2 | 角色: `ceo` \| 范围: `GLOBAL` |

---

### 5.4 施工日志（CONSTRUCTION_LOG）

> 施工日志是**独立系统**，不影响薪资结算，不纳入预结算校验。

**单条日志流程：**
```
[劳工在申报周期内提交日志]
    ↓ PENDING
[PM 审批：通过 / 驳回（可追加补充内容，不覆盖原文）]
    ↓ APPROVED → ARCHIVED
[CEO 追溯驳回（任意时刻）] → RECALLED（劳工须重新提交，原记录保留）
```

**工作项模板（ProjectLevel）：**
- PM 和 CEO 可配置工作项模板，供劳工填报时快速选择
- 可基于现有模板派生新模板；无模板时退回自由文本填写

**进度里程碑：**
- 只记录 `actualCompletionDate`，不设计划日期（工程预测偏差大）
- PM 拖动进度条标记完成，系统写入时间戳，CEO 实时可见

**汇总报告（PM 触发）：**
- 聚合指定项目+时间段内所有 APPROVED 日志
- PM 选择可视化组件（折线图/里程碑时间轴/工作项汇总表）+ 填写总体说明
- 自动 ARCHIVED，通知 CEO 查看

**申报周期配置：**
- 默认 DAILY（每天），可选 EVERY_2_DAYS / WEEKLY / CUSTOM
- **CEO 直接修改**，无需发起审批（写入 `Project.logReportCycleDays`）

---

### 5.5 薪资结算更正（PAYROLL_CORRECTION）

> 此流程**不是标准单据审批流**，而是 `PayrollCycle` 状态驱动的解锁流程。

**流程：**
```
[财务发起更正申请，填写更正原因]
    ↓ CORRECTION_PENDING → [CEO 审批]
APPROVED → 周期状态置为 UNLOCKED → 财务重新结算（新版本，版本号+1）
         → 新版本推送员工重新确认 → 历史版本保留，状态标记为 SUPERSEDED
REJECTED → 周期保持 LOCKED，财务可修改原因后再次发起
```

---

### 5.6 审批流通用约束

- 当前版本**不支持条件分支**（金额超阈值走不同节点）
- **不支持并行会签**
- 审批流变更仅对**变更后新提交**的单据生效，历史在途单据保持原流程配置
- 单据一旦进入 `ARCHIVED`，普通角色不可撤回；**CEO 拥有追溯驳回权限**
- 多 PM 项目中，**提交人在提交时从项目 PM 列表中选择一位**审批，用 `assignedReviewerId` 字段指定
- 所有审批节点操作（提交、审批、驳回、归档、追溯驳回）均写入 `operation_log`

## 6. 薪资规则配置

### 6.1 结算周期

| 参数 | 说明 | 默认值 | 可配置人 |
|-----|------|-------|---------|
| 周期类型 | 当前仅 `MONTHLY`（月结），P3 支持日结/周结 | `MONTHLY` | CEO |
| 起始日 | 每周期起始日 | 1日 | CEO |
| 发薪日 | 工资发放日，支持当月或次月 | 当月 25 日 | CEO |
| 结算截止日 | 财务须完成结算的日期 | 当月 25 日 | CEO |
| 结算提醒 | 结算截止日前 7 天，系统向财务推送提醒 | 系统自动 | — |

> 结算由财务**手动发起**，非自动触发。

---

### 6.2 预结算流程（窗口期模型）

窗口期字段内嵌于 `PayrollCycle`，**无独立 `PayrollWindowPeriod` 表**：

```
PayrollCycle.windowStatus: OPEN → 到期自动关闭 → CLOSED
```

**窗口期不可提前关闭**，只能由 `PayrollWindowScheduler`（每天 00:30）到期自动关闭。

**窗口期内各方操作权限：**

| 角色 | 可操作内容 |
|-----|-----------|
| 员工/劳工 | 查看本人考勤；提出考勤异议；提交请假申请（关闭后不再受理） |
| PM | 发起加班通知；审批日志；补充处理本期未完成的审批 |
| 财务 | 查看考勤数据全览；标记/修正异常考勤 |
| CEO | 查看全局状态；配置窗口期时长（`windowDays`） |

**窗口期结束后的数据锁定规则：**

| 数据类型 | 锁定处理 |
|---------|---------|
| 考勤异常（未处理） | 按"缺勤"处理，从当月工资中扣款 |
| 加班（未申报/未审批完成） | 不计入本月；审批跨月完成后计入审批完成当月 |
| 请假（窗口期内提交但审批未完成） | 审批完成后计入审批完成当月 |
| 请假（追溯补申报，任意时刻） | 审批完成后扣款计入审批完成当月 |
| 工伤补偿（审批未完成） | 不阻塞本月结算，理赔金额在任意后续周期录入 |

**结算前置校验（2 项强制，不可豁免）：**

| 序号 | 校验项 | 通过条件 |
|-----|--------|---------|
| 1 | 无未处理异议 | 无 `PENDING_REVIEW` 状态的 `PayrollSlip` |
| 2 | 无进行中计算任务 | 无 `CALCULATING` 状态的 `PayrollCycle` |

### 6.3 工资项构成与计算公式

| 工资项 | 计算类型 | 适用对象 | 规则 |
|-------|---------|---------|------|
| 基本工资 | 固定 | 全员 | 取员工等级 `baseSalary`，未配等级则取岗位 `baseSalary` |
| 绩效奖金 | 岗位固定值 | `hasPerformanceBonus=true` 的岗位 | 取岗位/等级 `performanceBase`；财务可设置个人覆盖值（需CEO审批） |
| 加班补贴 | 自动计算 | 全员 | 见下方公式 |
| 请假扣款 | 自动计算 | 全员 | 见下方公式 |
| 工伤补偿 | 手动录入 | LABOR | 财务在理赔完成后手动录入，关联至指定薪资周期（可跨月） |
| 可配置费目 | 手动挂载 | 按需 | 见§6.5 |
| 社保（个人部分） | 自动计算 | 全员 | 见§6.4 |
| 住房公积金（个人） | 自动计算 | 全员 | 基本工资 × 岗位 `housingFundRate`（默认 5%） |
| 个人所得税 | 自动计算 | 全员 | 按**累计预扣法**计算 |
| 实发工资 | 汇总 | 全员 | 基本工资 + 绩效 + 加班 + 工伤 + 补贴 − 请假扣款 − 社保 − 公积金 − 个税 − 扣减项 |

**加班补贴公式：**
```
日薪 = overtimeBase / 21.75   （overtimeBase 取值：BASIC_SALARY / TOTAL_SALARY / CUSTOM）
时薪 = 日薪 / 8
补贴 = 时薪 × 对应倍率 × 加班小时数
  平日倍率（overtimeRateWeekday）：默认 1.5
  周末倍率（overtimeRateWeekend）：默认 2.0
  节假日倍率（overtimeRateHoliday）：默认 3.0
```

**请假扣款公式：**
```
日薪 = leaveDeductBase / 21.75
扣款 = 日薪 × 请假天数 × LeaveTypeDef.deductRate
```

**假种扣款比例（LeaveTypeDef，可配置）：**

| 假种 | 默认 `deductRate` |
|-----|-----------------|
| 年假 | 1.0（全额扣） |
| 事假 | 1.0 |
| 病假 | 0.5 |
| 婚假 | 0.0（不扣薪） |
| 产假 | 0.0 |
| 其他 | 1.0 |

### 6.4 社保配置

**两种社保模式（`socialInsuranceMode`，可配置，CEO审批后下一周期生效）：**

| 模式 | 描述 | 工资条体现 |
|-----|------|----------|
| `COMPANY_PAID`（公司代缴） | 公司代缴，个人部分从实发中扣除 | 显示"社保扣款（个人）"，按险种分行展示 |
| `MERGED_SALARY`（并入工资） | 五险一金补贴合计并入工资；员工须自行办理**灵活就业**参保手续 | 显示"五险一金补贴（合计）"，不扣除 |

**社保险种默认比例（`SocialInsuranceItem` 表，可配置）：**

| 险种名 | 计算基数 | 默认个人比例 | 适用模式 |
|-------|---------|------------|---------|
| 养老保险 | 基本工资 | 8% | COMPANY_PAID |
| 医疗保险 | 基本工资 | 2% | COMPANY_PAID |
| 失业保险 | 基本工资 | 0.5% | COMPANY_PAID |
| 五险一金补贴 | 基本工资 | 10.5%（合计） | MERGED_SALARY |

## 7. 数据保留策略

### 7.1 默认保留期

> 所有数据默认保留 **1 年**，到期前 30 天提醒 CEO 备份。延长保留期功能为后续收费功能，当前版本不开放配置。

| 数据类型 | 默认保留期 | 说明 |
|---------|-----------|------|
| 请假记录 | 1 年 | — |
| 加班记录 | 1 年 | — |
| 工伤补偿记录 | 1 年 | 涉及法规合规，建议备份后延期 |
| 施工日志（单条） | 1 年 | — |
| 施工日志汇总报告 | 1 年 | — |
| 工资条（含历史版本） | 1 年 | 历史版本与最新版本同等对待 |
| 工资确认存证（含 PDF） | 1 年 | 建议在员工离职时手动备份 |
| 电子签名档案 | 1 年 | 涉及法律证明效力，建议备份后延期 |
| 审批历史记录 | 1 年 | — |
| 系统操作日志 | 1 年 | — |

### 7.2 到期处理规则

```
到期前 30 天：系统向 CEO 推送通知（待办角标 + 系统通知列表）
  通知内容：数据类型、预计到期日、数量

CEO 操作选项：
  a. 导出后删除 → 触发全量备份导出，导出完成后自动执行清理（推荐）
  b. 忽略       → 到期时自动执行清理，不再提醒
  （延期功能预留接口，当前 UI 不展示）

到期自动清理流程：
  定位到期业务记录 → 查找关联附件元数据
    → 删除附件物理文件
        ├─ 成功 → 删除数据库记录 → 写入审计日志
        └─ 失败 → 进入重试队列（最多 3 次，每次间隔 24h）
                   全部失败 → cleanup_task.status = FAILED
                            → Sysadmin 在 Admin 控制台「系统状态」Tab 处理
                   P3：连续 N 天未处理时，通过 SmsService 发送短信提醒 Sysadmin
```
