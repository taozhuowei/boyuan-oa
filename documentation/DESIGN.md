## 1. 组织架构与人员体系

### 1.1 概念模型

系统中的人员由四个**独立正交**的维度描述：

```
员工 Employee（系统的最小主体单元）
 ├── 岗位 Position      你是什么人、干什么活、怎么算薪资、有哪些功能开关
 ├── 等级 Level         你在这个岗位里处于哪个薪资段（可选）
 ├── 角色 Role          你在系统里能做哪些操作（权限控制）
 └── 直系领导 Supervisor 你向谁汇报、谁审批你的单据
```

| 混淆场景 | 正确理解 |
|---------|---------|
| "项目经理"既是岗位也是系统权限 | 岗位="项目经理"控制薪资/功能；角色=项目经理 控制审批权限，二者通常对齐但可以分离 |
| "劳工"是特殊员工类型 | 劳工只是开启了 `CONSTRUCTION_LOG` 功能开关的岗位，系统里所有人统一为"员工" |
| 薪资规则应该配到角色上 | 薪资规则配到岗位/等级，角色只控制操作权限 |

---

### 1.2 岗位配置（Position）

岗位是薪资规则、功能开关、假期配置的主要载体。字段定义见 [tech/ARCHITECTURE.md](tech/ARCHITECTURE.md)，以下为建筑工程版预置岗位（每个内置角色对应一个，其余岗位由 CEO 在运营期自行配置）：

| positionCode（岗位代码） | positionName（岗位名称） | defaultRoleCode（默认角色） | parentPositionCode（上级岗位） | positionFeatures（岗位功能开关） | performanceBase（绩效基数，元） |
|-------------|-------------|----------------|--------------|-----------------|-----------------|
| `CEO` | CEO | 最高管理者（CEO） | — | — | 0 |
| `FINANCE` | 财务 | 财务（finance） | `CEO` | — | 0 |
| `PM` | 项目经理 | 项目经理（project_manager） | `CEO` | — | 0 |
| `OFFICE_STAFF` | 办公人员 | 员工（employee） | `CEO` | — | 0 |
| `WORKER` | 劳工 | 劳工（worker） | `PM` | `CONSTRUCTION_LOG` | 0 |

> 其他岗位（如焊工、管工、设计师等）由 CEO 在初始化或运营期按需新建，继承上述预置结构。

**positionFeatures 可选值（可扩展）：**
- `CONSTRUCTION_LOG`：该岗位员工须定期提交施工日志

**等级示例（劳工）：**
```
岗位：劳工（baseSalary=5000）
  等级 L1（初级）：baseSalary=4500，annualLeaveDays=5
  等级 L2（中级）：baseSalary=5500，annualLeaveDays=7
  等级 L3（高级）：baseSalary=7000，annualLeaveDays=10
```

### 1.3 配置继承链（薪资规则取值优先级）

薪资规则、社保配置、假期配置等所有可配置项遵循以下优先级（高优先级覆盖低优先级）：

```
优先级 1（最低）：全局默认值
    ↓ 可覆盖
优先级 2：岗位配置
    ↓ 可覆盖
优先级 3：等级配置（等级存在时生效）
    ↓ 可覆盖
优先级 4（最高）：个人覆盖（需 CEO 审批）
```

| 层级 | 可修改人 | 是否需要审批 | 生效时机 |
|-----|---------|------------|---------|
| 全局默认值 | CEO | 无需审批 | 立即 |
| 岗位配置 | CEO / 财务 | 财务修改需 CEO 审批 | 下个结算周期 |
| 等级配置 | CEO / 财务 | 同上 | 下个结算周期 |
| 个人覆盖 | CEO / 财务 | 财务修改需 CEO 审批 | 下个结算周期 |

### 1.4 组织树路由规则

每个员工有唯一直系领导，构成树形汇报关系，根节点为 CEO。

**直系领导取值优先级（由高到低）：**
1. 员工档案中显式指定的 `directSupervisorId`（直系领导ID，最高优先级，用于特殊情况覆盖）
2. 员工所属项目的项目负责人（若员工有活跃项目归属）
3. 员工所属部门的负责人（`Department.headEmployeeId`，部门负责人ID）
4. 上级部门的负责人（沿部门树向上逐级查找）
5. CEO（兜底，所有路径失败时）

**组织树用途：**

| 用途 | 说明 |
|-----|------|
| 审批路由 | 请假/加班找直系领导初审，无法确定时自动上浮到 CEO |
| 数据范围 | 项目经理可查看本项目员工数据 |
| 通知推送 | 加班通知由 PM/CEO 直接发送给选中成员；工资条发放后全量通知所有在薪员工 |
| 汇报展示 | 经营总览的人员组织图 |

**无直系领导时的兜底规则：**

| 场景 | 节点1（初审）处理方式 |
|-----|---------------------|
| 员工无直系领导且无归属项目 | 自动转给 CEO |
| 员工归属项目但项目无 PM | 自动转给 CEO |

> CEO 执行了节点1（初审）后，节点2（终审）仍由 CEO 执行（CEO 可连续审批同一单据的两个节点，属于兜底设计）。

**初始化组织架构示例：**
```
CEO
 └── 财务（汇报给 CEO）
 └── 项目经理 A（汇报给 CEO）
      └── 现场工长（汇报给项目经理 A）
           └── 焊工 × N（汇报给现场工长）
 └── 项目经理 B
      └── ...
```

## 2. 角色定义

### 2.1 内置角色

| 角色名称 | roleCode | 定位 |
|---------|---------|------|
| 最高管理者 | `ceo` | 企业最高管理者，全局数据访问，系统配置，薪酬不在本系统内管理 |
| 财务 | `finance` | 薪资结算执行方，兼任人员档案管理（含新增/编辑），所有修改需 CEO 审批 |
| 项目经理 | `project_manager` | 项目范围内初审，施工日志审批与模板管理 |
| 员工 | `employee` | 普通办公室员工，发起个人申请，查看本人数据 |
| 劳工 | `worker` | 现场施工人员，额外有施工日志和工伤补偿提交权 |

**角色管理规则：**
- `ceo` 角色不可删除
- 其他内置角色及自定义角色均可删除、修改（删除前需先解除人员绑定）
- 角色的初始配置由系统管理员（admin）在初始化时敲定，运营期由 CEO 管理
- 每名员工当前版本只分配**一个主角色**
- 高风险权限（终审审批、解锁结算更正、配置审批流、配置数据保留策略）不可分配给非 CEO 角色

### 2.2 数据范围

| 角色 | 数据范围 | 说明 |
|-----|---------|------|
| 员工 | 本人 | 只能操作本人申请、档案、工资条 |
| 劳工 | 本人 | 同上；施工日志和工伤补偿限定于本人所属项目 |
| 项目经理 | 项目内 | 只作用于本人负责的项目及该项目下的员工 |
| 财务 | 全局 | 财务及人员数据全局可见；不含施工日志等非财务数据 |
| 最高管理者 | 全局 | 全局所有数据 |

## 4. 权限边界说明

### 4.1 终审规则

**双轨终审机制**：每条审批流在配置时独立指定是否设置 CEO 终审节点。

| 轨道 | 配置方式 | 终审判定 |
|-----|---------|---------|
| 需要 CEO 终审 | 审批流末节点配置为 CEO | CEO 审批通过 = 终审完成 |
| 不需要 CEO 终审 | 审批流末节点为非 CEO | 最后一个节点审批通过 = 终审完成 |

**无直系领导时的兜底**：当申请人无法确定直系领导时，当前节点自动转给 CEO。若 CEO 同时担任初审和终审节点，系统不拦截（兜底设计，允许同一人连续审批）。

---

### 4.2 工伤补偿的初审路由规则

工伤补偿节点1审批人依据**提交人身份**动态决定：

| 提交人 | 节点1 初审人 | 说明 |
|-------|------------|------|
| 劳工本人 | 项目经理（同项目） | 标准路径 |
| 项目经理（代录） | **跳过节点1**，直接进入 CEO 终审 | 提交人即初审人，自动跳过 |
| 财务（代录） | 项目经理（同被提交人项目） | 财务不参与现场核实，仍由项目经理初审 |
| 其他员工/劳工（代录） | 项目经理（同被提交人项目） | 路由同财务代录 |
| 无项目经理时（任意情形） | CEO（兜底） | — |

---

### 4.3 财务编辑角色权限的限制

`finance` 可编辑角色权限，但系统**不允许**为任何角色开启以下高风险权限：

- 终审审批
- 审批结算更正（解锁周期）
- 配置审批流
- 配置数据保留策略
- 新增/删除自定义角色

以上权限项在财务的角色管理页中显示为只读。

---

### 4.4 加班通知制的例外路径

所有员工不可自主申报加班，统一由 PM 或 CEO 发起通知。**例外路径**：当 PM 漏发通知时，员工可提交自补加班申请（附证明材料），走 PM → CEO 双审批。

| 场景 | 说明 |
|-----|------|
| PM 发起通知 → CEO 审批 | 标准路径，需 CEO 终审 |
| CEO 直接发起通知 | 自动归档，无需额外审批 |
| 员工自补申请（PM 漏报） | 需附证明材料，走 PM 初审 + CEO 终审 |
| 无归属项目的员工自补 | PM 节点自动转 CEO，由 CEO 单审 |

员工可在通知中标记"未参与"并说明原因，由 PM 最终确认是否接受拒绝。

---

### 4.5 施工日志提交前提

`worker` 角色员工提交施工日志时，必须已被分配到至少一个**进行中的项目**。若无项目归属，提交按钮灰显，提示"请联系管理员分配项目"。

---

### 4.6 运营期配置修改的审批约束

非 CEO 角色对以下配置的修改均需提交 CEO 审批，**下一个结算周期生效**：

| 配置类型 | 可发起修改的角色 | 生效时机 |
|---------|----------------|---------|
| 岗位配置（薪资/假期/社保规则） | finance | 下个结算周期 |
| 等级配置 | finance | 下个结算周期 |
| 员工个人薪资覆盖值 | finance | 下个结算周期 |
| 绩效奖金个人覆盖值 | finance | 下个结算周期 |
| 全局薪资规则（倍率/比例） | finance | 下个结算周期 |

CEO 直接修改以上任何配置均无需审批；社保模式切换需在系统中确认生效时机提示。

**配置变更通知**：岗位薪资、假期、等级等信息修改后，系统通知所有受影响员工。当前版本仅支持站内通知；短信和邮件通知为后续扩展功能（见 [tech/TODO.md](tech/TODO.md)）。

## 5. 审批流配置

### 5.1 请假申请（LEAVE）

**流程：**
```
[员工提交]
    ↓ PENDING
[节点1: 直系领导 初审，项目/部门范围；无直系领导时转 CEO]
    ↓ APPROVING
[节点2: CEO 终审，全局]
    ↓
APPROVED → ARCHIVED（自动纳入薪资结算扣款计算）
REJECTED → 申请人可查看驳回原因，重新提交（新单据，历史保留）
```

**配置参数：**

| 参数 | 值 |
|------|---|
| 业务类型 | 请假 |
| 节点数 | 2（默认，可配置） |
| 节点1 | 直系领导，项目或部门范围 |
| 节点2 | CEO（终审，可配置是否保留，见 [§4.1](#41-终审规则)） |
| 追溯申报 | 任意时刻可补申报；审批通过后扣款计入**审批完成当月**工资单 |

**表单字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|-----|------|
| 假种（leaveType） | 下拉 | ✓ | 年假/事假/病假/婚假/产假/其他 |
| 开始日期（startDate） | 日期 | ✓ | 追溯申报时可选历史日期 |
| 结束日期（endDate） | 日期 | ✓ | 不早于开始日期 |
| 请假天数（leaveDays） | 只读 | — | 系统自动计算，排除节假日和周末 |
| 申请原因（reason） | 文本 | ✓ | 最多 500 字 |
| 附件（attachments） | 文件 | 可选 | 最多 3 个，jpg/png/pdf，单文件 ≤ 10MB |
| 是否追溯（isRetroactive） | 只读 | — | 系统标记，追溯申报时为 true |

### 5.2 加班申请（OVERTIME）

> **加班采用通知制**：所有员工不可自主申报加班，统一由 PM 或 CEO 发起。

**PM 发起路径：**
```
[PM 发起加班通知（选项目/成员/日期/说明）]
    → 通知发送给选中员工，员工默认同意；可拒绝并说明原因，由 PM 终审
    ↓ PENDING → [CEO 终审] → APPROVED → ARCHIVED（计入薪资结算）
```

**CEO 直接发起：**
```
[CEO 发起加班通知] → 自动 APPROVED → ARCHIVED
```

**员工自补路径（PM 漏报时）：**
```
[员工补报，需上传证明材料] → PENDING → [PM 初审] → [CEO 终审] → APPROVED → ARCHIVED
```

> 加班跨月结算：审批完成后计入**审批完成当月**工资单，与请假追溯规则一致。

**配置参数：**

| 参数 | 值 |
|------|---|
| 业务类型 | 加班 |
| 发起方 | 项目经理 / CEO（员工自补路径除外） |
| CEO 发起 | 自动归档，无终审节点 |

**PM/CEO 发起通知表单字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|-----|------|
| 项目（projectId） | 下拉 | ✓ | — |
| 加班日期（overtimeDate） | 日期 | ✓ | 不晚于今日 |
| 加班类型（overtimeType） | 下拉 | ✓ | 平日/周末/节假日 |
| 加班小时数（hours） | 数字 | ✓ | 按最小单位起步，上限 24h |
| 目标员工（targetEmployees） | 多选 | ✓ | 默认选中项目全部在职成员 |
| 原因（reason） | 文本 | ✓ | 最多 500 字 |

**员工自补路径额外字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|-----|------|
| 证明材料（evidenceFiles） | 文件 | ✓ | 照片/截图，最多 5 个 |

### 5.3 工伤补偿（INJURY）

**代录规则**：任何员工可代他人提交工伤补偿申请，填写被提交人工号即可。

**初审路由**：见 §4.2。

**流程：**
```
[提交] → PENDING
[节点1: PM 初审（依路由规则，PM代录时跳过）]
    ↓ APPROVING
[节点2: CEO 终审]
    ↓
APPROVED → ARCHIVED（记录工伤事实，进入待理赔状态）
    → 财务在保险理赔完成后录入理赔金额，关联至目标薪资周期（可跨月）
REJECTED → 申请人可修改后重新提交
```

> 理赔金额录入不在申请表单中，由财务在工伤审批归档后、保险理赔结束时录入。

**配置参数：**

| 参数 | 值 |
|------|---|
| 业务类型 | 工伤补偿 |
| 节点1跳过条件 | 提交人为项目经理时自动跳过节点1 |
| 节点2 | CEO 终审（全局） |

**申请表单字段：**（理赔金额不在申请表单中，由财务在保险理赔完成后另行录入）

| 字段 | 类型 | 必填 | 说明 |
|------|------|-----|------|
| 受伤日期（injuryDate） | 日期 | ✓ | 不晚于今日 |
| 受伤部位（injuredPart） | 文本 | ✓ | 最多 100 字 |
| 受伤经过（description） | 文本 | ✓ | 最多 1000 字 |
| 医疗记录（medicalRecords） | 文件 | ✓ | 最多 10 个，jpg/png/pdf，单文件 ≤ 20MB |
| 代录员工（proxyEmployee） | 员工选择 | 代录时必填 | 填写被代录人工号 |

### 5.4 施工日志（CONSTRUCTION_LOG）

> 施工日志是**独立系统**，不影响薪资结算，不纳入预结算校验。

**单条日志流程：**
```
[劳工在申报周期内提交日志]
    ↓ PENDING
[PM 审批：通过 / 驳回（可追加补充内容，不覆盖原文）]
    ↓ APPROVED → ARCHIVED
[CEO 追溯驳回（任意时刻）] → RECALLED（劳工须重新提交，原记录保留）
```

**工作项模板（ProjectLevel）：**
- PM 和 CEO 可配置工作项模板，供劳工填报时快速选择
- 可基于现有模板派生新模板；无模板时退回自由文本填写

**进度里程碑：**
- 只记录 `actualCompletionDate`，不设计划日期（工程预测偏差大）
- PM 拖动进度条标记完成，系统写入时间戳，CEO 实时可见

**汇总报告（PM 触发）：**
- 聚合指定项目+时间段内所有 APPROVED 日志
- PM 选择可视化组件（折线图/里程碑时间轴/工作项汇总表）+ 填写总体说明
- 自动 ARCHIVED，通知 CEO 查看

**申报周期配置：**
- 默认每天，可选隔天 / 每周 / 自定义
- CEO 直接修改，无需发起审批

**提交表单字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|-----|------|
| 项目（projectId） | 下拉 | ✓ | 仅显示本人所属的进行中项目 |
| 日志日期（logDate） | 日期 | ✓ | 默认今日；同一项目同一日期不可重复 |
| 工作项（workItems） | 动态列表 | ✓ | 至少一行；每行：工作目标 + 数量 + 单位；优先从模板选择，可自定义 |
| 工作内容补充（workContent） | 文本 | 可选 | 最多 1000 字 |
| 现场照片（photos） | 图片 | 可选 | 最多 5 张，jpg/png，单张 ≤ 10MB |

### 5.5 薪资结算更正（PAYROLL_CORRECTION）

> 此流程**不是标准单据审批流**，而是 `PayrollCycle` 状态驱动的解锁流程。

**流程：**
```
[财务发起更正申请，填写更正原因]
    ↓ CORRECTION_PENDING → [CEO 审批]
APPROVED → 周期状态置为 UNLOCKED → 财务重新结算（新版本，版本号+1）
         → 新版本推送员工重新确认 → 历史版本保留，状态标记为 SUPERSEDED
REJECTED → 周期保持 LOCKED，财务可修改原因后再次发起
```

---

### 5.6 审批流通用约束

- 当前版本**不支持条件分支**（金额超阈值走不同节点）
- **不支持并行会签**
- 审批流变更仅对**变更后新提交**的单据生效，历史在途单据保持原流程配置
- 单据一旦进入 `ARCHIVED`，普通角色不可撤回；**CEO 拥有追溯驳回权限**
- 多 PM 项目中，**提交人在提交时从项目 PM 列表中选择一位**审批，用 `assignedReviewerId` 字段指定
- 所有审批节点操作（提交、审批、驳回、归档、追溯驳回）均写入 `operation_log`

## 6. 薪资规则配置

### 6.1 结算周期

| 参数 | 说明 | 默认值 | 可配置人 |
|-----|------|-------|---------|
| 周期类型 | 当前仅 `MONTHLY`（月结），P3 支持日结/周结 | `MONTHLY` | CEO |
| 起始日 | 每周期起始日 | 1日 | CEO |
| 发薪日 | 工资发放日，支持当月或次月 | 当月 25 日 | CEO |
| 结算截止日 | 财务须完成结算的日期 | 当月 25 日 | CEO |
| 结算提醒 | 结算截止日前 7 天，系统向财务推送提醒 | 系统自动 | — |

> 结算由财务**手动发起**，非自动触发。

---

### 6.2 预结算流程（窗口期模型）

窗口期字段内嵌于 `PayrollCycle`，**无独立 `PayrollWindowPeriod` 表**：

```
PayrollCycle.windowStatus（窗口期状态）: OPEN（开放）→ 到期自动关闭 → CLOSED（已关闭）
```

**窗口期不可提前关闭**，只能由 `PayrollWindowScheduler`（薪资窗口期调度器，每天 00:30）到期自动关闭。

**窗口期内各方操作权限：**

| 角色 | 可操作内容 |
|-----|-----------|
| 员工/劳工 | 查看本人考勤；提出考勤异议；提交请假申请（关闭后不再受理） |
| PM | 发起加班通知；审批日志；补充处理本期未完成的审批 |
| 财务 | 查看考勤数据全览；标记/修正异常考勤 |
| CEO | 查看全局状态；配置窗口期时长（`windowDays`，窗口期天数） |

**窗口期结束后的数据锁定规则：**

| 数据类型 | 锁定处理 |
|---------|---------|
| 考勤异常（未处理） | 按"缺勤"处理，从当月工资中扣款 |
| 加班（未申报/未审批完成） | 不计入本月；审批跨月完成后计入审批完成当月 |
| 请假（窗口期内提交但审批未完成） | 审批完成后计入审批完成当月 |
| 请假（追溯补申报，任意时刻） | 审批完成后扣款计入审批完成当月 |
| 工伤补偿（审批未完成） | 不阻塞本月结算，理赔金额在任意后续周期录入 |

**结算前置校验（2 项强制，不可豁免）：**

| 序号 | 校验项 | 通过条件 |
|-----|--------|---------|
| 1 | 无未处理异议 | 无 `PENDING_REVIEW`（待审核异议）状态的 `PayrollSlip`（工资条） |
| 2 | 无进行中计算任务 | 无 `CALCULATING`（计算中）状态的 `PayrollCycle`（薪资周期） |

### 6.3 工资项构成与计算公式

| 工资项 | 计算类型 | 适用对象 | 规则 |
|-------|---------|---------|------|
| 基本工资 | 固定 | 全员 | 取员工等级 `baseSalary`（基本工资额），未配等级则取岗位 `baseSalary` |
| 绩效奖金 | 岗位固定值 | `performanceBase`（绩效奖金基数）> 0 的岗位 | 取岗位/等级绩效基数；财务可设置个人覆盖值（需CEO审批）；配置为 0 则不发放 |
| 加班补贴 | 自动计算 | 全员 | 见下方公式 |
| 请假扣款 | 自动计算 | 全员 | 见下方公式 |
| 工伤补偿 | 手动录入 | 具有 `CONSTRUCTION_LOG` 功能开关的岗位员工 | 财务在理赔完成后手动录入，关联至指定薪资周期（可跨月） |
| 可配置费目 | 手动挂载 | 按需 | 见§6.5 |
| 社保（个人部分） | 自动计算 | 全员 | 见§6.4 |
| 住房公积金（个人） | 自动计算 | 全员 | 基本工资 × 岗位 `housingFundRate`（住房公积金比例，默认 5%） |
| 个人所得税 | 自动计算 | 全员 | 按**累计预扣法**计算 |
| 实发工资 | 汇总 | 全员 | 基本工资 + 绩效 + 加班 + 工伤 + 补贴 − 请假扣款 − 社保 − 公积金 − 个税 − 扣减项 |

**加班补贴公式：**
```
日薪 = overtimeBase（加班计算基数） / 21.75
       overtimeBase 取值：BASIC_SALARY（基本工资）/ TOTAL_SALARY（含绩效工资总额）/ CUSTOM（自定义金额）
时薪 = 日薪 / 8
补贴 = 时薪 × 对应倍率 × 加班小时数
  平日倍率（overtimeRateWeekday，平日加班倍率）：默认 1.5
  周末倍率（overtimeRateWeekend，周末加班倍率）：默认 2.0
  节假日倍率（overtimeRateHoliday，节假日加班倍率）：默认 3.0
```

**请假扣款公式：**
```
日薪 = leaveDeductBase（请假扣款基数） / 21.75
扣款 = 日薪 × 请假天数 × LeaveTypeDef.deductRate（假种扣款比例）
```

**假种扣款比例（LeaveTypeDef，假种定义表，可配置）：**

| 假种 | 默认 `deductRate`（扣款比例） |
|-----|-----------------|
| 年假 | 0.0（不扣薪）> **TODO**：未来可配置按比例扣除绩效奖金，如年假不扣基础工资但按天扣绩效 |
| 事假 | 1.0 |
| 病假 | 0.5 |
| 婚假 | 0.0（不扣薪） |
| 产假 | 0.0 |
| 其他 | 1.0 |

## 7. 数据保留策略

### 7.1 数据存储与保留期

以下数据存储于服务器本地数据库及文件系统（非用户设备）：

| 数据类型 | 默认保留期 | 说明 |
|---------|-----------|------|
| 请假记录 | 1 年 | — |
| 加班记录 | 1 年 | — |
| 工伤补偿记录 | 1 年 | 涉及法规合规，建议备份后延期 |
| 施工日志（单条） | 1 年 | — |
| 施工日志汇总报告 | 1 年 | — |
| 工资条（含历史版本） | 1 年 | — |
| 工资确认存证（含 PDF） | 1 年 | 建议员工离职时手动备份 |
| 电子签名档案 | 1 年 | 涉及法律证明效力，建议备份后延期 |
| 审批历史记录 | 1 年 | — |
| 系统操作日志 | 1 年 | — |

**保留期配置规则：**
- 系统管理员（admin）在初始化时设置各类型的默认保留期
- CEO 在运营期可按数据类型分别调整保留期（如单独延长工资条保留期）
- **托管版本**（非自建）：默认保留期上限为 1 年，延期需额外付费开启扩展功能
- **自建部署版本**：可自行配置，不受上限限制

### 7.2 到期处理规则

```
到期前 30 天：系统向 CEO 推送通知（待办角标 + 系统通知列表）
  通知内容：数据类型、预计到期日、数量

CEO 操作选项：
  a. 导出后删除 → 触发全量备份导出，导出完成后自动执行清理（推荐）
  b. 忽略       → 到期时自动执行清理，不再提醒
  （延期功能预留接口，当前 UI 不展示）

到期自动清理流程：
  定位到期业务记录 → 查找关联附件元数据
    → 删除附件物理文件
        ├─ 成功 → 删除数据库记录 → 写入审计日志
        └─ 失败 → 进入重试队列（最多 3 次，每次间隔 24h）
                   全部失败 → cleanup_task.status = FAILED
                            → Sysadmin 在 Admin 控制台「系统状态」Tab 处理
                   P3：连续 N 天未处理时，通过 SmsService 发送短信提醒 Sysadmin
```
