# 博渊 OA 平台 — 产品设计文档

> **文档职责**：描述博渊平台是什么、能做什么、设计原则是什么，以及第二家企业如何基于博渊接入自己的业务方案。
>
> **目标读者**：产品负责人、实施顾问、架构师。
>
> **不包含内容**：具体的业务配置（见 `presets/`）、代码实现细节（见 `platform/ARCHITECTURE.md` 和 `IMPLEMENTATION.md`）。

---

## 1. 平台定位

博渊是面向中小型工程与建筑类企业的**可配置 OA 协同管理平台**。

平台的核心设计理念是**引擎 + 配置分离**：平台提供通用的业务引擎（权限、工作流、表单、薪资、数据生命周期），企业通过配置驱动这些引擎，无需修改平台代码即可实现定制化的业务流程。

```
┌─────────────────────────────────────────────────────┐
│                   企业业务需求                         │
│  （角色定义 / 审批流 / 表单字段 / 薪资规则 / 模块选型）  │
└────────────────────────┬────────────────────────────┘
                         │ 配置输入
┌────────────────────────▼────────────────────────────┐
│                   博渊平台引擎层                       │
│  权限引擎 / 工作流引擎 / 表单引擎 / 薪资引擎           │
│  通知系统 / 签名存证 / 数据生命周期 / 文件管理          │
└────────────────────────┬────────────────────────────┘
                         │ 运行时渲染
┌────────────────────────▼────────────────────────────┐
│               双端应用（H5 + 小程序）                  │
└─────────────────────────────────────────────────────┘
```

## 2. 设计原则

**原则一：配置驱动，平台与业务解耦**
平台不内置具体业务逻辑（如"请假审批需要项目经理初审"）。所有业务规则以配置形式存储，平台引擎读取配置并执行，使同一套代码可以服务不同行业的企业。

**原则二：渐进式接入，开箱即用**
平台提供预置方案（`presets/`），企业可直接采用最接近自身需求的预置方案，经 Sysadmin 初始化向导快速上线，无需从零配置。

**原则三：系统管理与业务运营分层**
系统实施/运维人员（Sysadmin）与企业内部业务用户（CEO、财务等）使用独立的账号体系和操作界面，二者职责不交叉，确保平台稳定性不依赖业务用户的操作。

**原则四：存证与合规优先**
涉及薪资确认、审批记录等法律敏感操作，平台原生支持电子签名、操作留痕、数据完整性校验，满足劳动合同法相关合规要求。

## 3. 平台标准能力

### 3.1 身份与账号体系

| 能力          | 说明                                                       |
|--------------|-------------------------------------------------------------|
| 账号认证      | 工号 + 密码登录，JWT Bearer Token，支持企业微信 OAuth（可选）  |
| Sysadmin 层  | 系统实施账号，独立于业务用户，负责初始化和运维                   |
| 双端登录      | 同一账号可同时登录 Web 端和企业微信小程序                       |
| 密码安全      | bcrypt 加密存储，支持 Sysadmin 强制重置                        |

### 3.2 权限引擎

| 能力      | 说明                                                         |
|----------|--------------------------------------------------------------|
| 角色体系  | 支持多角色定义，角色数量不限，可设置系统内置（不可删）或自定义    |
| 权限粒度  | 菜单级、功能按钮级、数据范围级（全局 / 部门 / 项目 / 本人）      |
| 角色分配  | 每名员工分配一个主角色（当前版本），由 CEO / 财务在员工管理中配置  |
| 动态菜单  | 后端根据角色下发菜单树，前端配置驱动渲染，无硬编码路由权限         |

### 3.3 工作流引擎

| 能力      | 说明                                                          |
|----------|---------------------------------------------------------------|
| 审批流定义 | 每种业务类型对应一个审批流，由有序节点列表组成                    |
| 节点配置  | 审批人可按角色或指定人配置，节点数量不限                          |
| 状态机    | `DRAFT → PENDING → APPROVING → APPROVED/REJECTED → ARCHIVED`  |
| 驳回处理  | 驳回后原单据保留历史，申请人可重新发起                            |
| 变更隔离  | 审批流变更仅对新提交单据生效，不影响在途单据                       |

### 3.4 表单引擎

| 能力      | 说明                                               |
|----------|----------------------------------------------------|
| 表单配置  | 字段类型、必填规则、选项列表、校验规则均可配置          |
| 模板管理  | 支持按业务类型定义表单模板，支持版本化                  |
| 配置下发  | 前端按需请求表单配置，Session 内缓存，后端版本变更时自动清除 |

### 3.5 薪资引擎

| 能力      | 说明                                                       |
|----------|------------------------------------------------------------|
| 工资项配置 | 基本工资、绩效、加班、扣款、社保、公积金、个税等项目可配置       |
| 结算周期  | 支持月结 / 半月结，可配置                                     |
| 多版本结算 | 更正申请触发新版本结算，历史版本保留                            |
| 状态流转  | `PREVIEW → PENDING_CONFIRM → CONFIRMED/DISPUTED → ARCHIVED` |

### 3.6 电子签名与存证

| 能力         | 说明                                                        |
|-------------|-------------------------------------------------------------|
| 手写签名     | Canvas 手写录入，加密存档，绑定员工 ID                         |
| 独立 PIN 码  | 签名使用独立 PIN（与登录密码分离），bcrypt 加密                  |
| 证据链       | 每次签署生成完整证据链（时间、IP、设备、内容哈希、签名哈希）        |
| 存证扩展     | 预留 e签宝 CA 接入能力，通过配置项切换，不影响业务逻辑             |
| 保留期       | 存证 PDF 保留不少于劳动合同终止后 5 年                          |

### 3.7 通知系统

| 能力         | 说明                                            |
|-------------|------------------------------------------------|
| 系统内通知   | 工作台待办角标、系统通知列表                       |
| 企业微信推送  | 审批通知、工资条发布、到期提醒（需企业微信接入配置）   |

### 3.8 数据生命周期管理

| 能力      | 说明                                                   |
|----------|---------------------------------------------------------|
| 保留策略  | 按数据类型配置保留期，支持全局默认 + 按类别覆盖              |
| 到期提醒  | 到期前主动通知 CEO，支持延期或额外导出                      |
| 自动清理  | 到期自动删除数据库记录 + 附件物理文件，附件失败则进入重试队列   |
| 审计日志  | 每次清理操作记录审计日志                                   |

### 3.9 文件管理

| 能力           | 说明                                                                                    |
|---------------|-----------------------------------------------------------------------------------------|
| 存储方案       | 本地文件系统存原始文件，数据库存附件元数据                                                    |
| 路径规范       | `/uploads/{业务类型}/{yyyy-MM}/{UUID}.{ext}`                                              |
| 支持的业务类型  | `leave / overtime / injury / construction-log / payroll-confirm / signature / export`   |

### 3.10 双端支持

| 终端                           | 定位                               |
|-------------------------------|-----------------------------------|
| 企业微信工作台小程序             | 移动办公、高频审批、轻量录入          |
| Web 端（PC / 平板 / 手机浏览器） | 配置管理、批量处理、复杂查询、报表     |

## 4. 可配置边界

### 4.1 可配置项（实施阶段通过 Sysadmin 向导或后台配置）

| 配置项         | 配置粒度    | 运行时可修改          |
|---------------|------------|----------------------|
| 角色定义与权限  | 角色级      | ✓                    |
| 审批流节点     | 业务类型级   | ✓（仅影响新单据）      |
| 表单字段       | 表单模板级   | ✓                    |
| 薪资计算规则   | 工资项级    | ✓（仅影响新周期）      |
| 数据保留策略   | 数据类型级   | ✓                    |
| 结算周期类型   | 全局        | ✓                    |
| 启用的功能模块  | 模块级      | ✓                    |
| 公司基本信息   | 全局        | ✓                    |

### 4.2 平台固定项（不可通过配置修改）

- JWT 认证机制与 Token 格式
- Sysadmin 账号层级与初始化流程
- 电子签名与存证的数据结构
- 文件存储路径规范
- 数据清理的物理删除机制
- 双端适配架构（uni-app + 组件适配层）
- 状态机流转规则（如审批流的基本状态）

## 5. 第二家企业接入指南

博渊的设计支持为不同行业的企业快速创建定制化方案，接入流程如下：

### 5.1 创建预置方案

在 `presets/` 下新建目录，命名建议使用行业英文名（如 `manufacturing`、`hvac`、`real-estate`）：

```
presets/
├── construction/     ← 建筑工程版（当前）
└── {新行业}/
    ├── BUSINESS.md           # 业务需求设计
    ├── ROLE_CONFIG.md        # 角色权限配置
    ├── WORKFLOW_CONFIG.md    # 工作流与业务规则
    └── UI_DESIGN.md          # 业务模块页面设计
```

### 5.2 配置决策清单

接入新企业时，需要与客户确认并文档化以下配置：

| 决策点           | 说明                        | 参考示例                                       |
|-----------------|-----------------------------|-------------------------------------------------|
| 启用哪些业务模块  | 考勤 / 薪资 / 项目管理等是否启用 | 建筑版：全启用                               |
| 角色定义         | 员工的职能分类，3～8 个角色为宜 | 建筑版：5 个角色                              |
| 权限矩阵         | 每个角色可访问的功能          | 见 `presets/construction/ROLE_CONFIG.md`        |
| 审批流节点       | 各业务类型的审批层级与审批人    | 建筑版：初审 + 终审                             |
| 薪资规则         | 工资项组成与计算公式          | 见 `presets/construction/WORKFLOW_CONFIG.md`    |
| 数据保留期       | 各类数据的保留时长             | 建筑版：按类型配置                               |

### 5.3 Sysadmin 初始化上线流程

```
1. 部署博渊平台服务
2. Sysadmin 账号通过初始化向导完成：
   a. 公司基本信息
   b. 创建第一个 CEO 账号
   c. 确认角色权限模板（基于预置方案的默认值）
   d. 确认审批流配置
   e. 设置数据保留策略默认值
3. CEO 登录，完成人员档案录入、角色分配
4. 员工凭工号 + 初始密码登录，完成首次登录设置
```

### 5.4 不同行业接入的差异点

| 差异维度       | 建筑工程版           | 其他行业版本示例          |
|--------------|---------------------|--------------------------|
| 核心业务模块   | 施工日志、工伤补偿    | 制造业：生产工单、设备巡检  |
| 特殊角色      | 劳工（劳务人员）      | 制造业：班组长             |
| 薪资规则复杂度  | 工伤补偿纳入薪资      | 制造业：计件工资            |
| 审批链长度     | 2 节点（初审+终审）   | 大型企业：3-4 节点          |

## 6. 版本规划

当前为 **v1.0 MVP 版本**，面向建筑工程行业的首个预置方案。

后续迭代方向（不作为当前开发范围）：
- 多角色支持（一名员工拥有多个角色）
- 条件分支审批（金额超过阈值走不同节点）
- 并行会签
- 多租户 SaaS 模式（一套部署服务多家企业）
- 移动端原生应用（React Native / Flutter）
