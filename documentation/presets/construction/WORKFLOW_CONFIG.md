# 博渊·建筑工程版 — 工作流与业务规则配置

> **文档职责**：定义建筑工程版各业务类型的审批流、状态机、薪资计算规则、数据保留策略和表单字段。
>
> **目标读者**：产品经理、后端开发者（种子数据与引擎配置参考）。
>
> **不包含**：工作流引擎实现原理（见 `platform/ARCHITECTURE.md §5.2`）；角色定义与初审路由规则（见 `ROLE_CONFIG.md §4`）。

---

## 1. 审批流配置

### 1.1 请假申请（LEAVE）

**流程：**
```
[员工提交]
    ↓ PENDING
[节点1: project_manager 初审，项目范围内]
    ↓ APPROVING
[节点2: CEO 终审，全局]
    ↓
APPROVED ──→ ARCHIVED（归档，自动纳入薪资结算扣款计算）
REJECTED ──→ 申请人可修改后重新提交（新单据，保留历史）
```

**状态机：**
```
DRAFT → PENDING → APPROVING → APPROVED → ARCHIVED
                       ↓
                   REJECTED（可重新发起）
```

**配置参数：**

| 参数          | 值                                                            |
|-------------|---------------------------------------------------------------|
| `businessType` | `LEAVE`                                                    |
| 节点数        | 2                                                             |
| 节点1         | 角色: `project_manager` \| 范围: `PROJECT` \| 类型: `ROLE`    |
| 节点2         | 角色: `ceo` \| 范围: `GLOBAL` \| 类型: `ROLE`                 |
| 驳回处理      | 单据状态置为 `REJECTED`，原单据完整保留，申请人可提交新单据       |
| 已归档单据    | 不可撤回，不可重新审批（CEO 追溯驳回权限见 §1.6）                  |

---

### 1.2 加班申请（OVERTIME）

与请假申请流程和节点配置完全相同。

**配置参数：**

| 参数          | 值                                                         |
|-------------|-------------------------------------------------------------|
| `businessType` | `OVERTIME`                                               |
| 节点1         | 角色: `project_manager` \| 范围: `PROJECT` \| 类型: `ROLE` |
| 节点2         | 角色: `ceo` \| 范围: `GLOBAL` \| 类型: `ROLE`              |

---

### 1.3 工伤补偿（INJURY）

**代录规则：** 任何员工均可代他人提交工伤补偿申请，无部门或项目限制，填写被提交人工号即可。代录人身份影响节点1路由（见下），但代录人本身不参与审批流程。

**初审路由规则：** 节点1 的审批人依据提交人动态确定（详见 `ROLE_CONFIG.md §4.2`）：
- worker 本人提交 → 节点1: 同项目 project_manager
- project_manager 代录 → **跳过节点1**，直接进入节点2（CEO）
- finance 代录 → 节点1: 同项目 project_manager（同被提交人所属项目）
- 其他员工/劳工代录 → 节点1: 同项目 project_manager（同被提交人所属项目）

**流程：**
```
[提交]
    ↓ PENDING
[节点1: project_manager 初审（依路由规则，可被跳过）]
    ↓ APPROVING
[节点2: CEO 终审]
    ↓
APPROVED ──→ ARCHIVED（记录工伤事实，进入待理赔状态）
                ↓（异步，保险理赔完成后）
            财务录入理赔金额，关联至目标薪资周期（可跨月）
REJECTED ──→ 申请人可修改后重新提交
```

> **理赔金额不在申请表单中填写**，而是由财务在工伤审批归档后、保险理赔结束时录入，并指定计入哪个薪资周期。理赔录入与审批归档可以不在同一月份。

**配置参数：**

| 参数            | 值                                                             |
|---------------|----------------------------------------------------------------|
| `businessType`  | `INJURY`                                                     |
| 节点1           | 动态（见路由规则），范围: `PROJECT`                               |
| 节点2           | 角色: `ceo` \| 范围: `GLOBAL` \| 类型: `ROLE`                  |
| 薪资联动         | 财务手动录入理赔金额后写入指定周期 `PayrollSlip.injury` 字段       |
| 节点1跳过条件    | 提交人 role == `project_manager`                               |

---

### 1.4 施工日志（CONSTRUCTION_LOG）

> **施工日志是独立系统**，不影响薪资结算，与审批流的接口仅限于：劳工提交日志 → 项目经理审核归档。

**流程：**
```
[劳工提交]
    ↓ PENDING
[节点1: project_manager 审批，项目范围内]
    ↓
APPROVED ──→ ARCHIVED
REJECTED ──→ 劳工可修改后重新提交

[CEO 追溯驳回（任意时刻）]
ARCHIVED ──→ RECALLED
RECALLED ──→ 劳工须重新提交（生成新版本，原记录完整保留）
```

**状态机：**
```
DRAFT → PENDING → APPROVED → ARCHIVED ←─┐
                      ↓                  │ CEO追溯驳回
                  REJECTED          RECALLED
               （可重新发起）
```

> **CEO 追溯驳回**：CEO 可在任何时刻对已归档的施工日志发起驳回，日志状态变为 `RECALLED`，劳工收到通知后须重新提交。此操作写入审计日志。

**配置参数：**

| 参数          | 值                                                         |
|-------------|-------------------------------------------------------------|
| `businessType` | `CONSTRUCTION_LOG`                                       |
| 节点数        | 1                                                           |
| 节点1         | 角色: `project_manager` \| 范围: `PROJECT` \| 类型: `ROLE` |

---

### 1.5 薪资结算更正（PAYROLL_CORRECTION）

> 此流程**不是标准单据审批流**，而是一个结算周期解锁流程，由 `PayrollCycle` 状态驱动。

**流程：**
```
[财务发起更正申请，填写更正原因]
    ↓ CORRECTION_PENDING
[CEO 审批]
    ↓
APPROVED:
  周期状态置为 UNLOCKED
  → 财务重新执行结算（生成新版本，版本号 +1）
  → 新版本推送员工重新确认
  → 历史版本完整保留，状态标记为 SUPERSEDED，不可篡改

REJECTED:
  周期保持 LOCKED 状态，更正申请记录驳回原因
  财务可修改原因后再次发起
```

**工资单多版本规则：**
- 每次重新结算生成新的 `PayrollSlip`，版本号递增（v1, v2, v3…）
- 员工只能看到并确认**最新版本**
- 历史版本只读，不接受确认或异议操作
- 所有版本均计入审计日志

---

### 1.6 审批流通用约束

- 当前版本**不支持条件分支**（如金额超过阈值走不同节点）
- **不支持并行会签**
- 审批流变更仅对**变更后新提交**的单据生效，历史在途单据保持原流程配置
- 单据一旦进入 `ARCHIVED`，普通角色不可撤回，**CEO 拥有追溯驳回权限**（适用于所有业务类型，施工日志进入 `RECALLED` 状态，其他类型视实际业务需求扩展）
- **所有审批节点操作**（提交、审批、驳回、归档、追溯驳回）均写入操作日志落盘，永久保留，不受数据保留策略影响

## 2. 薪资规则配置

### 2.1 结算周期

以下所有参数均由 sysadmin（初始化时）或 CEO（运营期）在系统配置页调整。

| 参数       | 说明                              | 初始默认值     |
|-----------|----------------------------------|--------------|
| 周期类型   | `MONTHLY`（月结）/ `BIMONTHLY`（半月结）| `MONTHLY` |
| 起始日     | 每周期起始日（如每月第1日）           | 1日           |
| 结束日     | 每周期结束日（如每月最后一日）         | 月末           |
| 发薪日     | 工资发放日，支持次月（如次月10日）      | 次月10日       |
| 结算截止日 | 财务须完成结算的日期（如当月25日）      | 当月25日       |
| 结算提醒   | 结算截止日前 **7 天**，系统向财务推送提醒 | 系统自动       |

> 结算触发由财务**手动发起**，非自动触发；到期提醒仅提醒，不自动执行。

---

### 2.2 工资项构成与计算公式

| 工资项              | 计算类型   | 适用对象     | 计算公式 / 规则                                                          |
|--------------------|----------|-------------|-------------------------------------------------------------------------|
| 基本工资            | 固定       | 全员         | 按员工所属薪资档位取值（见下方"薪资档位"说明）                                |
| 绩效奖金            | 手动录入   | OFFICE 类型  | 财务按周期手动录入，不自动计算                                               |
| 加班补贴            | 自动计算   | 全员         | 见下方公式                                                                |
| 请假扣款            | 自动计算   | 全员         | 见下方公式                                                                |
| 工伤补偿            | 手动录入   | LABOR 类型   | 财务在理赔完成后手动录入，关联至指定薪资周期（可跨月）                          |
| 社保（个人部分）     | 自动计算   | 全员         | 见下方"社保模式"说明                                                        |
| 住房公积金（个人）   | 自动计算   | 全员         | 基本工资 × 公积金比例（系统配置，默认 5%）                                   |
| 个人所得税          | 自动计算   | 全员         | 按**累计预扣法**计算，由平台引擎执行                                         |
| 实发工资            | 汇总      | 全员         | 基本工资 + 绩效奖金 + 加班补贴 + 工伤补偿 − 请假扣款 − 社保 − 公积金 − 个税  |

**薪资档位：**

sysadmin 初始化时按岗位/职级批量配置薪资档位（`SalaryGrade`），员工档案直接关联档位，无需逐人录入基本工资。CEO 和财务可在运营期调整档位或为个别员工单独覆盖。

```
SalaryGrade:
  gradeCode: 档位编码（如 A1, B2, LABOR_S1）
  gradeName: 档位名称（如 技术工人一级）
  baseSalary: 月固定工资（元）
```

**加班补贴公式：**
```
日薪 = 基本工资 / 21.75
时薪 = 日薪 / 8

平日加班补贴  = 时薪 × 1.5 × 加班小时数
周末加班补贴  = 时薪 × 2.0 × 加班小时数
节假日加班补贴 = 时薪 × 3.0 × 加班小时数
```

**请假扣款公式：**
```
日薪 = 基本工资 / 21.75

年假 / 事假 / 婚假 / 产假扣款 = 日薪 × 请假天数
病假扣款                      = 日薪 × 请假天数 × 0.5
```

**社保模式（可配置，初始化时选择）：**

| 模式         | 描述                                                                    | 工资条体现                             |
|------------|------------------------------------------------------------------------|--------------------------------------|
| **公司代缴** | 社保由公司代为缴纳，个人部分从实发工资中扣除                                  | 显示"社保扣款（个人）"，从实发中扣除     |
| **并入工资** | 社保金额并入基本工资发放，员工自行缴纳；工资条体现该部分为"社保补贴"              | 显示"社保补贴"，不从实发中扣除           |

**社保比例配置（两种计算方式可选）：**
```
方式A — 按险种分项配置（推荐）：
  养老保险：基本工资 × 8%（默认）
  医疗保险：基本工资 × 2%（默认）
  失业保险：基本工资 × 0.5%（默认）
  合计比例可按城市标准在系统配置中调整

方式B — 按总额配置：
  社保合计 = 基本工资 × 总比例（如 10.5%）
  适用于不需要分项展示的场景
```

> **工伤保险、生育保险**：由用人单位（企业）全额缴纳，不从员工工资中扣除，不计入此工资单。

---

### 2.3 预结算校验（4 项，全部通过方可正式结算）

| 序号 | 校验项      | 通过条件                                                        |
|-----|------------|----------------------------------------------------------------|
| 1   | 考勤完整性  | 周期内所有员工的考勤数据已录入，无未处理异常天数                    |
| 2   | 审批完整性  | 周期内所有请假 / 加班单据状态不为 `PENDING` 或 `APPROVING`        |
| 3   | 工伤完整性  | 周期内所有工伤补偿申请状态不为 `PENDING` 或 `APPROVING`           |
| 4   | 档案完整性  | 所有在职员工的薪资档案已配置（薪资档位不为空）                       |

> **施工日志不纳入预结算校验**：施工日志为独立系统，不影响薪资结算流程。

任一校验未通过，财务可查看具体未通过的员工或单据列表，修复后重新发起预结算。

## 3. 数据保留策略

### 3.1 默认保留期

> 所有数据默认保留 **1 年**，到期前 30 天提醒 CEO 备份。延长保留期功能作为后续收费点预留，当前不对外开放。

| 数据类型                  | 默认保留期 | 说明                              |
|--------------------------|----------|----------------------------------|
| 请假记录                  | 1 年     | —                                |
| 加班记录                  | 1 年     | —                                |
| 工伤补偿记录               | 1 年     | 涉及法规合规，建议备份后延期          |
| 施工日志                  | 1 年     | —                                |
| 工资条                    | 1 年     | —                                |
| 工资确认存证（含 PDF）      | 1 年     | 建议在员工离职时手动备份             |
| 电子签名档案               | 1 年     | 涉及法律证明效力，建议备份后延期      |
| 审批历史记录               | 1 年     | —                                |
| 系统操作日志               | 永久保留  | 操作日志不受保留策略管理，不可删除    |

> **操作日志例外**：§1.6 中定义的所有审批操作日志永久保留，不受上述保留策略影响。

### 3.2 到期处理规则

```
到期前 30 天：
  系统向 CEO 推送通知（待办角标 + 系统通知列表）
  通知内容：数据类型、预计到期日、数量

CEO 操作选项：
  a. 导出后删除 → 触发全量备份导出，导出完成后自动执行清理（推荐）
  b. 忽略       → 到期时自动执行清理，不再提醒

  注：延期功能预留接口，当前 UI 不展示（后续收费功能）

到期自动清理流程：
  定位到期业务记录
    → 查找关联附件元数据（AttachmentMeta）
    → 删除附件物理文件
        ├─ 成功 → 删除数据库记录 → 写入审计日志
        └─ 失败 → 进入重试队列（最多重试 3 次，每次间隔 24h）
                   多次失败 → 标记为"待人工处理"，通知 CEO
```

## 4. 表单字段定义

### 4.1 请假申请（LEAVE）

**最小计量单位可配置**（系统级配置，初始化时设定）：`小时` / `半天` / `天`，影响 `leaveDays` 的精度和扣款计算。

| 字段名          | 类型     | 必填  | 选项 / 约束                                    |
|---------------|---------|------|------------------------------------------------|
| `leaveType`   | 下拉选择  | ✓    | 年假 / 事假 / 病假 / 婚假 / 产假 / 其他           |
| `startDate`   | 日期选择  | ✓    | 不早于今日                                      |
| `endDate`     | 日期选择  | ✓    | 不早于 `startDate`                              |
| `leaveDays`   | 只读     | —    | 系统按配置的最小单位自动计算（排除法定节假日和周末）  |
| `reason`      | 文本域   | ✓    | 最多 500 字                                     |
| `attachments` | 文件上传  | 可选  | 最多 3 个；支持 jpg / png / pdf；单文件 ≤ 10 MB  |

### 4.2 加班申请（OVERTIME）

**最小计量单位与请假一致**（系统级配置），影响 `hours` 的精度和加班补贴计算。

| 字段名            | 类型     | 必填      | 选项 / 约束                                    |
|----------------|---------|----------|------------------------------------------------|
| `overtimeDate`  | 日期选择  | ✓        | 不晚于今日（不允许提前申请未来加班）                |
| `hours`         | 数字输入  | ✓        | 按系统配置的最小单位起步，上限 24 小时             |
| `overtimeType`  | 下拉选择  | ✓        | 平日加班 / 周末加班 / 节假日加班                  |
| `reason`        | 文本域   | ✓        | 最多 500 字                                     |
| `proxyEmployee` | 员工选择  | 代录时必填 | 仅 project_manager / ceo 可见此字段             |

### 4.3 工伤补偿（INJURY）

> 申请表单不包含工伤类型和理赔金额，这两项由财务在保险理赔完成后另行录入（见 §1.3 流程说明）。

| 字段名             | 类型     | 必填      | 选项 / 约束                                      |
|-----------------|---------|----------|--------------------------------------------------|
| `injuryDate`    | 日期选择  | ✓        | 不晚于今日                                        |
| `injuredPart`   | 文本输入  | ✓        | 最多 100 字                                       |
| `description`   | 文本域   | ✓        | 最多 1000 字                                      |
| `medicalRecords`| 文件上传  | ✓        | 最多 10 个；支持 jpg / png / pdf；单文件 ≤ 20 MB  |
| `proxyEmployee` | 员工选择  | 代录时必填 | 代录时对所有员工可见；填写被代录人工号                |

### 4.4 施工日志（CONSTRUCTION_LOG）

> 施工日志为独立系统，以下字段为提交审批流时的核心字段。

**工作项快捷录入（`workItems`）：** 动态列表，每行由员工在表单中自由添加，无需预先配置模板。系统根据列表自动生成结构化摘要（表格 + 文字描述）。

| 字段名          | 类型          | 必填  | 选项 / 约束                                          |
|--------------|-------------|------|------------------------------------------------------|
| `projectId`   | 下拉选择       | ✓    | 仅列出本人所属的进行中项目                               |
| `logDate`     | 日期选择       | ✓    | 默认今日；不晚于今日；同一项目同一日期不可重复提交          |
| `workItems`   | 动态列表       | ✓    | 至少填写一行；每行包含：`目标名称（文字）`+ `数量（数字）`+ `单位（文字）`；例：焊接口 / 8 / 个 |
| `workContent` | 文本域        | 可选  | 补充说明，最多 1000 字；系统将 `workItems` 和本字段合并输出最终日志 |
| `photos`      | 图片上传       | 可选  | 最多 5 张；支持 jpg / png；单张 ≤ 10 MB               |

> **系统输出规则**：提交后系统将 `workItems` 渲染为表格（目标 / 数量 / 单位），并追加 `workContent` 文字，合并为最终施工日志存档内容。
