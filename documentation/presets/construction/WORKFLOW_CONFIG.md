## 1. 审批流配置

### 1.1 请假申请（LEAVE）

**流程：**
```
[员工提交]
    ↓ PENDING
[节点1: project_manager 初审，项目范围内]
    ↓ APPROVING
[节点2: CEO 终审，全局]
    ↓
APPROVED ──→ ARCHIVED（归档，自动纳入薪资结算扣款计算）
REJECTED ──→ 申请人可修改后重新提交（新单据，保留历史）
```

**状态机：**
```
DRAFT → PENDING → APPROVING → APPROVED → ARCHIVED
                       ↓
                   REJECTED（可重新发起）
```

**配置参数：**

| 参数          | 值                                                            |
|-------------|---------------------------------------------------------------|
| `businessType` | `LEAVE`                                                    |
| 节点数        | 2                                                             |
| 节点1         | 角色: `project_manager` \| 范围: `PROJECT` \| 类型: `ROLE`    |
| 节点2         | 角色: `ceo` \| 范围: `GLOBAL` \| 类型: `ROLE`                 |
| 驳回处理      | 单据状态置为 `REJECTED`，原单据完整保留，申请人可提交新单据       |
| 已归档单据    | 不可撤回，不可重新审批                                           |

| 节点1         | 角色: `project_manager` \| 范围: `PROJECT` \| 类型: `ROLE` |
| 节点2         | 角色: `ceo` \| 范围: `GLOBAL` \| 类型: `ROLE`              |

### 1.3 工伤补偿（INJURY）

**初审路由规则：** 节点1 的审批人依据提交人动态确定（详见 `ROLE_CONFIG.md §4.2`）：
- worker 本人提交 → 节点1: 同项目 project_manager
- project_manager 代录 → **跳过节点1**，直接进入节点2（CEO）
- finance 代录 → 节点1: 同项目 project_manager

[COMMENT] 这里改成允许其他员工帮忙代提交，输入被提交人的工号就可以代替被提交人走流程，代提交人不能是项目经理，必须是其他同级员工，不规定是否必须同部门或同项目。

**流程（标准路径）：**
```
[提交]
    ↓ PENDING
[节点1: project_manager 初审（依路由规则，可被跳过）]
    ↓ APPROVING
[节点2: CEO 终审]
    ↓
APPROVED ──→ ARCHIVED（compensationAmount 自动写入当月工资单 injury 项）
REJECTED ──→ 申请人可修改后重新提交
```

**配置参数：**

| 参数            | 值                                                             |
|---------------|----------------------------------------------------------------|
| `businessType`  | `INJURY`                                                     |
| 节点1           | 动态（见路由规则），范围: `PROJECT`                               |
| 节点2           | 角色: `ceo` \| 范围: `GLOBAL` \| 类型: `ROLE`                  |
| 薪资联动         | 通过后将 `compensationAmount` 写入 `PayrollSlip.injury` 字段    |
| 节点1跳过条件    | 提交人 role == `project_manager`                               |

### 1.4 施工日志（CONSTRUCTION_LOG）

**流程：**
```
[劳工提交]
    ↓ PENDING
[节点1: project_manager 审批，项目范围内]
    ↓
APPROVED ──→ ARCHIVED
REJECTED ──→ 劳工可修改后重新提交
```

**状态机：**
```
DRAFT → PENDING → APPROVED → ARCHIVED
                      ↓
                  REJECTED（可重新发起）
```

> **无终审节点**：施工日志不进入 CEO 审批，由项目经理一级审核后即归档。CEO 不可干预此流程。

[COMMENT] 不需要CEO立即审核才能归档。但是归档后CEO有权限在任何时刻将施工日志驳回重写。

**配置参数：**

| 参数          | 值                                                         |
|-------------|-------------------------------------------------------------|
| `businessType` | `CONSTRUCTION_LOG`                                       |
| 节点数        | 1                                                           |
| 节点1         | 角色: `project_manager` \| 范围: `PROJECT` \| 类型: `ROLE` |

### 1.5 薪资结算更正（PAYROLL_CORRECTION）

> 此流程**不是标准单据审批流**，而是一个结算周期解锁流程，由 `PayrollCycle` 状态驱动。

**流程：**
```
[财务发起更正申请，填写更正原因]
    ↓ CORRECTION_PENDING
[CEO 审批]
    ↓
APPROVED:
  周期状态置为 UNLOCKED
  → 财务重新执行结算（生成新版本，版本号 +1）
  → 新版本推送员工重新确认
  → 历史版本完整保留，状态标记为 SUPERSEDED，不可篡改

REJECTED:
  周期保持 LOCKED 状态，更正申请记录驳回原因
  财务可修改原因后再次发起
```
[COMMENT] 工资单流程首先得是财务发起工资结算，然后系统自动拉取数据计算每个人的工资列表，财务可以将考勤数据、工伤申报等与最终工资强相关且每个月都会变的数据发布给每个员工进行初审，对数据没有异议后财务进行复核，然后推送最终工资条给每个人，然后员工签字确认，没问题就留档，有问题就上报，CEO解开权限后进行修改，更新版本继续审批直到没有问题。中间流程记录操作日志。

**工资单多版本规则：**
- 每次重新结算生成新的 `PayrollSlip`，版本号递增（v1, v2, v3…）
- 员工只能看到并确认**最新版本**
- 历史版本只读，不接受确认或异议操作
- 所有版本均计入审计日志

---

### 1.6 审批流通用约束

- 当前版本**不支持条件分支**（如金额超过阈值走不同节点）
- **不支持并行会签**
- 审批流变更仅对**变更后新提交**的单据生效，历史在途单据保持原流程配置
- 单据一旦进入 `ARCHIVED`，任何角色**不可撤回、不可重新审批**
  
[COMMENT] CEO有权限接触已归档的状态将流程驳回重新提交。另外所有流程都要留操作日志落盘记录。

## 2. 薪资规则配置

### 2.1 结算周期

| 参数      | 值                       |
|----------|--------------------------|
| 周期类型  | 月结（`MONTHLY`）          |
| 自然月对齐 | 每月1日至月末              |
| 发薪日    | 次月10日（可配置）          |
| 结算触发  | 财务**手动发起**，非自动触发  |

[COMMENT] 周期、月起始日和结束日、发薪日（支持次月发薪）都可以由admin和CEO配置。结算需要额外配置结算日，到期前一周提醒财务进行结算操作。
---

### 2.2 工资项构成与计算公式

| 工资项              | 计算类型   | 适用对象     | 计算公式 / 规则                                                          |
|--------------------|----------|-------------|-------------------------------------------------------------------------|
| 基本工资            | 固定       | 全员         | 员工薪资档案中配置的月固定工资，直接取值                                      |
| 绩效奖金            | 手动录入   | OFFICE 类型  | 财务按周期手动录入，不自动计算                                               |
| 加班补贴            | 自动计算   | 全员         | 见下方公式                                                                |
| 请假扣款            | 自动计算   | 全员         | 见下方公式                                                                |
| 工伤补偿            | 申请金额   | LABOR 类型   | 当月已通过的工伤补偿申请 `compensationAmount` 直接汇总                       |
| 社保（个人部分）     | 自动计算   | 全员         | 见下方公式                                                                |
| 住房公积金（个人）   | 自动计算   | 全员         | 基本工资 × 公积金比例（员工档案配置，默认 5%）                               |
| 个人所得税          | 自动计算   | 全员         | 按**累计预扣法**计算，由平台引擎执行                                         |
| 实发工资            | 汇总      | 全员         | 基本工资 + 绩效奖金 + 加班补贴 + 工伤补偿 − 请假扣款 − 社保 − 公积金 − 个税  |

**加班补贴公式：**
```
日薪 = 基本工资 / 21.75
时薪 = 日薪 / 8

平日加班补贴  = 时薪 × 1.5 × 加班小时数
周末加班补贴  = 时薪 × 2.0 × 加班小时数
节假日加班补贴 = 时薪 × 3.0 × 加班小时数
```

**请假扣款公式：**
```
日薪 = 基本工资 / 21.75

年假 / 事假 / 婚假 / 产假扣款 = 日薪 × 请假天数
病假扣款                      = 日薪 × 请假天数 × 0.5
```

**社保（个人部分）公式：**
```
默认比例（可在系统配置中按城市标准调整）：
  养老保险：基本工资 × 8%
  医疗保险：基本工资 × 2%
  失业保险：基本工资 × 0.5%
  
个人社保合计 = 基本工资 × (8% + 2% + 0.5%) = 基本工资 × 10.5%
```

> **工伤保险、生育保险**：由用人单位（企业）全额缴纳，不从员工工资中扣除，不计入此工资单。


[COMMENT] 固定工资在初始化阶段配置不同岗位的薪资档位，避免一个一个录入。社保默认不是公司交，而是把这部分钱直接并入工资，这是可配置的，初始化要给两个选项，社保是公司交还是并入工资，可以配置各个部分的比例，或者直接配置一个总额，取固定工资的百分比。如果是

### 2.3 预结算校验（5 项，全部通过方可正式结算）

| 序号 | 校验项      | 通过条件                                                        |
|-----|------------|----------------------------------------------------------------|
| 1   | 考勤完整性  | 周期内所有员工的考勤数据已录入，无未处理异常天数                    |
| 2   | 审批完整性  | 周期内所有请假 / 加班单据状态不为 `PENDING` 或 `APPROVING`        |
| 3   | 日志完整性  | 周期内所有劳工（LABOR 类型）的施工日志状态不为 `PENDING`           |
| 4   | 工伤完整性  | 周期内所有工伤补偿申请状态不为 `PENDING` 或 `APPROVING`           |
| 5   | 档案完整性  | 所有在职员工的薪资档案已配置（基本工资、社保比例不为空）              |

任一校验未通过，财务可查看具体未通过的员工或单据列表，修复后重新发起预结算。

## 3. 数据保留策略

### 3.1 默认保留期

> 以下为 Sysadmin 初始化时写入的系统默认值，CEO 可在数据管理页修改（受限制项除外）。

| 数据类型                  | 默认保留期          | 是否允许缩短 | 说明                     |
|--------------------------|-------------------|------------|--------------------------|
| 请假记录                  | 5 年              | ✓          | —                        |
| 加班记录                  | 5 年              | ✓          | —                        |
| 工伤补偿记录               | 永久              | ✗          | 涉及工伤保险法规合规要求    |
| 施工日志                  | 3 年              | ✓          | —                        |
| 工资条                    | 5 年              | ✓          | —                        |
| 工资确认存证（含 PDF）      | 劳动合同终止后 5 年 | ✗          | 计时起点为员工离职日期      |
| 电子签名档案               | 永久              | ✗          | 签名档案不允许删除          |
| 审批历史记录               | 5 年              | ✓          | —                        |
| 系统操作日志               | 2 年              | ✓          | —                        |

### 3.2 到期处理规则

```
到期前 30 天：
  系统向 CEO 推送通知（待办角标 + 系统通知列表）
  通知内容：数据类型、预计到期日、数量

CEO 操作选项：
  a. 延期  → 指定延期时长（天/月/年），更新到期时间
  b. 导出后删除 → 触发全量备份导出，导出完成后自动执行清理
  c. 忽略  → 到期时自动执行清理，不再提醒

到期自动清理流程：
  定位到期业务记录
    → 查找关联附件元数据（AttachmentMeta）
    → 删除附件物理文件
        ├─ 成功 → 删除数据库记录 → 写入审计日志
        └─ 失败 → 进入重试队列（最多重试 3 次，每次间隔 24h）
                   多次失败 → 标记为"待人工处理"，通知 CEO
```

## 4. 表单字段定义

### 4.1 请假申请（LEAVE）

| 字段名          | 类型     | 必填  | 选项 / 约束                                    |
|---------------|---------|------|------------------------------------------------|
| `leaveType`   | 下拉选择  | ✓    | 年假 / 事假 / 病假 / 婚假 / 产假 / 其他           |
| `startDate`   | 日期选择  | ✓    | 不早于今日                                      |
| `endDate`     | 日期选择  | ✓    | 不早于 `startDate`                              |
| `leaveDays`   | 只读     | —    | 系统自动计算（排除法定节假日和周末）                |
| `reason`      | 文本域   | ✓    | 最多 500 字                                     |
| `attachments` | 文件上传  | 可选  | 最多 3 个；支持 jpg / png / pdf；单文件 ≤ 10 MB  |

### 4.2 加班申请（OVERTIME）

| 字段名            | 类型     | 必填      | 选项 / 约束                               |
|----------------|---------|----------|------------------------------------------|
| `overtimeDate`  | 日期选择  | ✓        | 不晚于今日（不允许提前申请未来加班）          |
| `hours`         | 数字输入  | ✓        | 0.5 起步，步长 0.5，上限 24               |
| `overtimeType`  | 下拉选择  | ✓        | 平日加班 / 周末加班 / 节假日加班             |
| `reason`        | 文本域   | ✓        | 最多 500 字                               |
| `proxyEmployee` | 员工选择  | 代录时必填 | 仅 project_manager / ceo 可见此字段       |

### 4.3 工伤补偿（INJURY）

| 字段名                 | 类型     | 必填      | 选项 / 约束                                      |
|---------------------|---------|----------|--------------------------------------------------|
| `injuryDate`         | 日期选择  | ✓        | 不晚于今日                                        |
| `injuredPart`        | 文本输入  | ✓        | 最多 100 字                                       |
| `injuryType`         | 下拉选择  | ✓        | 轻伤 / 重伤 / 骨折 / 其他                          |
| `description`        | 文本域   | ✓        | 最多 1000 字                                      |
| `medicalRecords`     | 文件上传  | ✓        | 最多 10 个；支持 jpg / png / pdf；单文件 ≤ 20 MB  |
| `compensationAmount` | 数字输入  | ✓        | 单位：元，精度：2 位小数，最小值 0.01               |
| `proxyEmployee`      | 员工选择  | 代录时必填 | 仅 project_manager / finance / ceo 可见此字段     |

### 4.4 施工日志（CONSTRUCTION_LOG）

| 字段名          | 类型     | 必填  | 选项 / 约束                                          |
|--------------|---------|------|------------------------------------------------------|
| `projectId`   | 下拉选择  | ✓    | 仅列出本人所属的进行中项目                               |
| `logDate`     | 日期选择  | ✓    | 默认今日；不晚于今日；同一项目同一日期不可重复提交          |
| `workContent` | 文本域   | ✓    | 最多 1000 字                                          |
| `workHours`   | 数字输入  | ✓    | 0.5 起步，步长 0.5，上限 24                           |
| `photos`      | 图片上传  | 可选  | 最多 5 张；支持 jpg / png；单张 ≤ 10 MB               |
