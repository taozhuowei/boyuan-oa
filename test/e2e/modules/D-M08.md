## 文档约定

- 本文件是 `test/e2e/specs/D-M08.spec.ts` 的实现依据，按"页面/步骤 → 操作/组件 → 子检查"三层组织
- 每个加粗组件名对应一个 Playwright `test()`，子检查以 `test.step()` 实现
- 禁止使用 `test.skip()` 或条件 skip；所有用例必须真实执行并拿到 Pass/Fail 结果
- 所有断言必须含前端可见反馈（Toast / 红框 / 跳转 / DOM 变化），禁止仅调 API
- 跨步骤数据流必须断言：上一步填的数据在下一步显示 / 提交体内出现

## 前置 hook 约定

- `test.beforeAll`：`POST /dev/reset` 清业务数据 + `POST /dev/reset-setup` 重置初始化状态
- `test.beforeEach`：`POST /dev/reset-setup` + `POST /dev/reset-finalize` 让向导回到 step 1 起点（系统未初始化态）
- `test.afterAll`：`POST /dev/skip-setup` 标记系统已初始化避免影响其他模块
- 鉴权令牌测试前：通过 init 提交后从响应捕获 `wizardFinalizeToken`

## 1. 路由守卫与初始化状态

### 1.1 系统未初始化访问 /setup
- 直接访问 `/setup`，URL 保持 `/setup`，渲染 step 1（CEO）
- 无需登录态

### 1.2 系统已初始化但向导未完成（/setup/init 已成功，/setup/finalize 未成功）
- 模拟：先调 init 拿响应，但不调 finalize
- 访问 `/setup`，URL 保持 `/setup`，仍可看到 step 5 / 6 / ... 等任意步骤（路由守卫不踢人）

### 1.3 系统已初始化且向导已完成
- 模拟：先调 init + finalize 全成功
- 访问 `/setup`，URL 自动跳转到 `/login`

### 1.4 GET /setup/status 端点
- 公开访问，无 token 也返回 200
- 未初始化：`{initialized:false, wizardFinalizeCompleted:false, companyName:null}`
- init 后未 finalize：`{initialized:true, wizardFinalizeCompleted:false, companyName:<填写值>}`
- finalize 后：`{initialized:true, wizardFinalizeCompleted:true, companyName:<填写值>}`

## 2. Step 1 — CEO 账号表单

### 2.1 企业名称字段（可选）
- 留空可继续
- 填写后在 step 4 确认页 "企业名称" 行显示该值

### 2.2 CEO 姓名字段
- 空提交触发 Toast"请输入CEO姓名"，无网络请求
- 超长 200 字符被 maxlength 截断或 Toast 提示
- XSS payload `<script>alert(1)</script>` 不触发 alert，正常作为字符串保存

### 2.3 CEO 手机号字段
- 空 → Toast"请输入CEO手机号"
- 非 1[3-9]\d{9} 格式（如 12345678901）→ Toast"请输入有效的手机号码"
- 合法 13800001111 被接受

### 2.4 CEO 密码字段
- 空 → Toast"请输入密码"
- 长度 < 8 → Toast"密码长度须为 8-64 位"
- 长度 > 64 → Toast"密码长度须为 8-64 位"
- 无字母（如 12345678）→ Toast"密码须包含字母"
- 无数字（如 abcdefgh）→ Toast"密码须包含数字"
- 含空格（如 `abc 12345`）→ Toast"密码不能包含空格"
- 合法（如 Secret123）→ 强度提示三项变绿（长度 / 字母 / 数字）

### 2.5 确认密码字段
- 与密码不一致 → Toast"两次输入的密码不一致"
- 一致 → 通过

### 2.6 下一步按钮
- 全部合法 + 点"下一步" → 进入 step 2，已填字段保留

## 5. Step 4 — 确认页 + 提交（POST /setup/init）

### 5.1 信息展示
- a-descriptions 显示 CEO 姓名 / CEO 手机号 / HR 姓名 / HR 手机号 / 运维 / 总经理 6 行
- 未填的运维 / 总经理显示"-"
- 企业名称如填则显示，未填显示默认"博渊OA管理系统"

### 5.2 上一步按钮
- 点"上一步" → 回 step 3，运维/总经理字段保留

### 5.3 提交成功路径
- 点"提交" → 按钮 loading 状态
- 网络请求 `POST /setup/init` 状态码 200
- 响应包含 `recoveryCode`（32 字符）+ `wizardFinalizeToken`（32 字符）+ `message`
- UI 自动进入 step 5，显示恢复码

### 5.4 提交失败：系统已初始化（403）
- 模拟：先成功 init 一次，再访问 step 4 提交
- 实际不可能直接到达此 case（路由守卫已踢），但可手动调 API 验证返回 403

### 5.5 提交失败：字段缺失（400）
- 模拟：手动构造缺 ceoName 的请求体调 API
- 返回 400 + 错误消息，前端 step 4 显示 a-alert 错误提示

### 5.6 提交失败：唯一性冲突（409）
- 模拟：构造 ceoPhone 与 hrPhone 相同的请求
- 返回 409 + 错误消息

## 9. Step 8 — 汇报关系（SupervisorTree）

### 9.1 组件加载
- 显示 CEO 节点固定在树根
- 显示 step 7 导入的员工列表（左侧未分配区）

### 9.2 拖拽员工到 CEO 节点
- 拖拽员工 A 到 CEO 节点下 → 树更新，员工 A 出现在 CEO 子节点
- supervisors 数组新增 `{employeeTempId: A.tempId, supervisorTempId: 'ceo'}`

### 9.3 拖拽员工到员工节点（多级汇报）
- 员工 A 已在 CEO 下
- 拖拽员工 B 到员工 A 下 → 树正确显示 B 是 A 的下属

### 9.4 循环检测
- 员工 A 在 CEO 下，员工 B 在 A 下
- 拖拽 A 到 B 下 → invalid 事件触发，Toast 提示循环错误，操作被拦截

### 9.5 CEO 不可移动
- 拖拽 CEO 节点 → invalid 事件 reason='CEO_IMMUTABLE'，Toast 提示

### 9.6 拖到节点间隙
- 拖动到非节点的间隙位置 → invalid reason='DROP_TO_GAP'，Toast 提示

### 9.7 跳过按钮 / 下一步
- 跳过 → supervisors 清空 → step 9
- 下一步 → 数据保留 → step 9

## 11. Step 10 — 数据保留期（RetentionPanel） + 完成初始化（POST /setup/finalize）

### 11.1 RetentionPanel 加载
- 表格显示当前策略（默认空或全 3 年）
- "新增保留策略"按钮 + "编辑"/"删除"操作列

### 11.2 新增保留策略
- 点"新增" → 模态框：dataType select / retentionDays / earlyWarningDays
- 提交 → 表格新增一行

### 11.3 编辑 / 删除
- 编辑 → 模态框回显 + 修改 + 保存
- 删除 → confirm + 移除

### 11.4 完成初始化按钮
- 点"完成初始化" → 按钮 loading 状态
- POST /setup/finalize 网络请求

### 11.5 finalize 提交体内容
- 抓取请求 body，验证字段：
  - `wizardFinalizeToken` 是 init 返回的值
  - `roles` 是 step 6 收集的数组
  - `employeeImport.departments / positions / employees` 是 step 7 收集
  - `organization.supervisors` 是 step 8 收集（含 tempId）
  - `globalConfig.entries` 是 step 9 收集
  - `approvalFlows` 是 step 9 收集
  - `retention.policies` 是 step 10 收集

### 11.6 finalize 成功路径
- 响应 200 + `{message: "初始化向导完成"}`
- Toast"系统初始化完成"
- 自动跳转到 `/login`

### 11.7 finalize 失败：401 token 错误
- 模拟：手动构造缺 token 或错误 token 的请求
- 响应 401 + 错误消息
- 前端 Toast 提示

### 11.8 finalize 失败：409 已 finalize
- 模拟：先成功 finalize 一次，再调一次
- 响应 409 + "初始化已完成，不能重复调用"
- Toast 提示

### 11.9 finalize 失败：400 字段错误
- 模拟：构造 roles[0].code=null 的请求
- 响应 400 + 错误消息
- @Transactional 整体回滚（验证：再调 GET /api/roles 看不到任何 finalize 时的角色）

### 11.10 finalize 部分失败回滚
- 构造 roles 含一条合法 + 一条非法
- 期望 4xx + 整体回滚
- 验证：合法 roles[0] 也未写入数据库

## 14. /setup/finalize 端点单独测试（API 层）

### 14.1 系统未 init 时调用
- 未 init → 400"系统未初始化，请先完成基础账号创建"

### 14.2 token 缺失
- 不带 wizardFinalizeToken → 400 字段校验失败

### 14.3 token 错误
- 带错误 token → 401"finalize 令牌不正确"

### 14.4 已 finalize 重复调用
- 第一次成功后再调 → 409"初始化已完成，不能重复调用"

### 14.5 字段全部为 null
- 构造 `{wizardFinalizeToken: <valid>, roles: null, employeeImport: null, ..., retention: null}` → 200 成功（用户全部跳过）

### 14.6 部分字段填写部分跳过
- roles: [...合法] + 其他 null → 200 + 仅角色被创建

### 14.7 角色 code 与系统内置冲突
- roles[0].code = "ceo" → 400"自定义角色编码 ceo 与系统内置角色冲突"

### 14.8 supervisor tempId 引用不存在的员工
- supervisors[0].employeeTempId = "emp-999"（员工列表无此项）→ 400"未知的员工 tempId: emp-999"

### 14.9 审批流 businessType 无效
- approvalFlows[0].businessType = "INVALID" → 400"无效的审批流业务类型: INVALID"

### 14.10 字段 fail-fast（P1-3 修复）
- approvalFlows[0].businessType = "" → 400 + 字段路径"approvalFlows[0].businessType 为空"
- 不静默跳过

### 14.11 finalize 成功后 GET /setup/status
- finalize 成功 → status.wizardFinalizeCompleted = true

### 14.12 finalize 成功后 GET /api/roles
- finalize 成功后 + CEO 登录 → 看到 finalize 创建的自定义角色（验证 AccessManagementService 双写注入）

## 15. /dev 辅助端点（测试基础设施）
