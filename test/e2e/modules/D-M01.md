# D-M01 认证模块 E2E 测试用例清单

用户确认日期：2026-04-23

## 文档约定

- 本文件是 `test/e2e/specs/D-M01.spec.ts` 的实现依据，按"页面 → 组件/操作 → 子检查"三层组织
- 每个加粗组件名对应一个 Playwright `test()`，子检查以 `test.step()` 实现
- 禁止使用 `test.skip()` 或条件 skip；所有用例必须真实执行并拿到 Pass/Fail 结果
- 所有断言必须含前端可见反馈（Toast / 红框 / 跳转 / DOM 变化），禁止仅调 API

## 前置 hook 约定

- `test.beforeAll`：`POST /dev/reset` 清业务数据
- `test.beforeEach`（限流测试除外）：`POST /dev/reset-rate-limit` 清限流计数
- CEO 恢复码测试前：`POST /dev/reset-setup`；测试结束：`POST /dev/skip-setup`
- 首次登录设置测试前：通过 CEO 权限 API 恢复 employee.demo 为默认密码状态

## 登录页（/login）

### 用户名输入框

- 空值提交被 Ant Design Form 拦截，未触发 /auth/login 网络请求
- 超长 1000 字符输入被 maxlength 截断或提交返回错误
- XSS payload `<script>alert(1)</script>` 不触发 alert，显示错误提示
- SQL 注入 `admin' OR '1'='1` 无绕过，显示错误提示
- 合法值 `employee.demo` 被接受无前端错误

### 密码输入框

- 空值提交被拦截，无网络请求
- 超长 200 字符被 maxlength 截断或提交返回错误
- XSS payload 不触发 alert
- SQL 注入无绕过
- 合法值被接受

### 登录提交行为

- 正确账密登录成功 → URL 变为 /workbench，导航含角色名
- 错误密码停留 /login，显示"账号或密码错误"提示
- 不存在账号停留 /login，显示错误提示
- 被禁账号（`worker.demo` 先由 CEO 禁用）登录失败，显示禁用提示
- 首次登录默认密码（employee.demo/123456）重定向 /setup-account

### 图形验证码挑战

- 清限流后前 3 次输错密码，响应不含 `captchaRequired`
- 第 4 次提交响应 `captchaRequired: true`，页面出现验证码图片元素
- 不填验证码提交返回 400 + `captchaRequired: true`
- 填错验证码提交返回 400 + `captchaRequired: true`
- 点击验证码图片 → 触发 GET /auth/captcha → 图片 src base64 发生变化
- 填对验证码 + 正确密码登录成功跳 /workbench
- GET /auth/captcha 独立调用返回 `{captchaId, imageBase64}` 格式正确

### per-IP 限流

- 本测试不调 reset-rate-limit
- 对不同账号连续 5 次错密（跨账号以排除 per-account 早锁）
- 第 6 次返 429，响应含 `message: 登录尝试过于频繁`、`selfServiceUnlock` 字段
- 调 `POST /dev/reset-rate-limit` 后用正确账密再次登录成功

### per-account 限流

- 调 reset-rate-limit 清零
- 对同账号 `employee.demo` 用错密码连续提交 5 次 → 第 6 次返 429 锁定该账号
- 同 IP 另一账号 `worker.demo` 正确密码仍可登录（per-IP 未锁，per-account 仅锁 employee.demo）
- 恢复 employee.demo 密码为 123456（测试收尾）

### 自助解锁

- 调 reset-rate-limit 清零
- 用错密码连续锁定 employee.demo
- 锁定响应 body 含 `selfServiceUnlock: /me/forgot-password`
- 走忘记密码流程（/auth/forgot_password）重置 employee.demo 密码为新密码
- 立即用新密码登录成功，不需要等锁超时
- 收尾：恢复 employee.demo 密码为 123456

### 登录后跳转

- 非首次普通账号（非默认密码）跳 /workbench
- 首次账号（默认密码 123456）跳 /setup-account
- 登录 CEO、HR、Finance、PM、Employee、Worker、DeptManager 各角色，都能进入各自的工作台（URL /workbench，导航含对应菜单）

## 首次登录设置页（/setup-account）

### 邮箱输入框

- 空值提交被前端表单校验拦截
- 非法格式被拦：`plainaddress`、`@no-local-part.com`、`no-at-sign.com`、`a@b@c.com`、`no-domain@`
- 带 tag `user+tag@example.com` 被接受
- 超长 300 字符被 maxlength 截断或提交返回错误
- 已被他人占用的邮箱（`finance.demo` 的邮箱）后端返回 400 + 占用提示
- 合法自己邮箱 → 发码按钮可点

### 发送验证码按钮

- 邮箱为空时按钮禁用或点击后拦截
- 合法邮箱点击 → 成功 Toast + 按钮进入 60s 倒计时（文本形如 "60s 后重发"）
- 冷却期间再次点击按钮禁用无网络请求
- 调 `GET /dev/verification-code?type=bind&email=xxx` 能读到缓存码

### 验证码输入框

- 空值提交被前端校验拦截
- 输入字母 `abcdef`：字段按规则处理（仅留数字或报错，视前端实现）
- XSS payload 不触发 alert
- 超长 100 字符被 maxlength=6 截断至 6
- 错码提交显示"验证码错误"
- 正确码（从 dev API 获取）进入下一步

### 新密码输入框

- 输入纯字母 `abcdefgh`，"包含数字"提示红色状态
- 输入少于 8 字符 `Ab1`，"长度 8-64"提示红色状态
- 输入缺大写 `abcdef123`，对应强度指示红色
- 输入合格 `Abcd1234`，所有强度指示绿色

### 确认密码输入框

- 与新密码不一致提示错误，提交按钮禁用或被拦
- 一致通过校验

### 提交行为

- 提交成功 → 后端 employee.email 写入新绑定邮箱，isDefaultPassword 变为 false
- 跳转 /workbench
- 退出后用旧默认密码登录失败
- 用新设置的密码登录成功
- 收尾：通过 CEO API 恢复 employee.demo 为默认密码状态

- 点击发送验证码

- 收尾：恢复 employee.demo 密码为 123456

## 路由守卫

### 未登录访问保护页

- 浏览器无 oa-token cookie
- 直接访问 /workbench → 重定向 /login

### 无效 token 访问

- 手工设置 oa-token cookie 值为 `invalid.jwt.token`
- 访问 /workbench → 后端返回 401 → 前端重定向 /login

### 无权限角色访问

- employee.demo 登录
- 访问仅 HR/CEO 可用的 /employees 页面
- 被路由守卫拦截，重定向回允许的页面（/workbench 或 /）

## CEO 恢复码（初始化向导）

### 向导显示恢复码

- 调 `POST /dev/reset-setup` 重置初始化状态
- 访问 /setup，走完向导 step 1 到 5（填公司信息、CEO 信息、管理员信息、模板选择等）
- step 5 页面显示 16 位恢复码（格式 `XXXX-XXXX-XXXX-XXXX` 或 16 位连写，视设计）
- 向导完成后调 `POST /dev/skip-setup` 恢复

### 复制按钮

- step 5 页面存在复制按钮
- 点击复制按钮 → clipboard 内容为该 16 位恢复码 + 页面显示"已复制"类 Toast

### 恢复码重置密码

- 事前记录或复制 step 5 的恢复码
- 在登录页点击"忘记密码"或直接进入 CEO 恢复码重置入口
- 填入 CEO 账号 + 恢复码 + 新密码
- 提交成功
- 用 CEO 账号 + 新密码登录成功
- 用 CEO 账号 + 旧密码登录失败
- 自助解锁验证：重置前先锁定 CEO 账号，恢复码重置成功后立即登录成功
- 收尾：将 CEO 密码恢复为 123456

## 实现指导

### 测试文件组织

- 单一 spec 文件：`test/e2e/specs/D-M01.spec.ts`
- 使用 `test.describe.serial` 保证顺序（部分用例有数据依赖）
- 每个粗体组件名对应一个 `test('<组件/操作名>', async ({ browser }) => {...})`
- 每个子检查用 `await test.step('<子检查描述>', async () => {...})`

### 工具依赖

- `test/tools/helpers/email-reader.ts` 的 `getVerificationCodeFromDev` 用于拿验证码
- `test/tools/fixtures/auth.ts` 的 `loginAs` 用于快速登录
- 限流 / 初始化状态通过新增或已有 `/dev/*` 端点重置
- 自助解锁测试的账号锁定复用真实的错密码连续提交逻辑（per-account）

### 断言原则

- 断言必须包含前端可见反馈（文本、URL、DOM 属性、数据标记）
- 使用 `page.getByTestId(...)` 定位（`data-catch` 属性，由 Playwright testIdAttribute 配置）
- 禁止依赖 `waitForTimeout` 作为同步手段，用 `waitFor` / `waitForURL` / `waitForResponse`

### 运行方式

- 本地：`E2E_BASE_URL=http://localhost:3001 npx playwright test --config test/e2e/playwright.config.ts test/e2e/specs/D-M01.spec.ts --reporter=line`
- 前置要求：H5 dev 服务（port 3001）+ 后端 dev profile（port 8080）+ PostgreSQL oa_dev 库
