**基准：** 2026-04-17 已执行测试（Round 1 + Round 2）  
**对标标准：** 等价类划分、边界值分析、状态转换测试、场景测试（ISTQB 基础级）

## 一、已覆盖的测试类型

- 路由守卫正向验证（7个角色 × 全部菜单页面）
- 安全边界（SQL注入、XSS、未认证访问、越权访问）

## 二、缺口清单

### 2.1 业务流程端到端（E2E Flows）—— 完全未覆盖

当前测试只验证了页面能否打开，没有任何一条完整的业务主干流程被测试。

#### 核心审批链
- 员工提交请假申请 → 部门经理审批通过 → 考勤记录生效
- 员工提交加班申报 → 审批通过 → 工资单中产生加班补贴
- 员工提交报销申请 → 财务审批 → 报销金额注入当月工资单
- 工人提交施工日志 → PM 审核 → 里程碑进度更新
- 工人提交工伤申报 → 财务录入理赔 → 审批通过
- 申请被驳回 → 员工修改后重新提交 → 再次审批

#### 薪资发放主链
- 财务创建薪资周期 → 开放申报窗口 → 员工提交考勤 → 窗口关闭 → 财务结算 → CEO 审核 → 员工签收
- 员工对工资单提出异议 → 财务发起更正 → CEO 审批 → 重新结算

#### 项目管理链
- CEO 创建项目 + 添加里程碑 → PM 填写施工日志 → 进度推进 → 里程碑完成 → 财务记录营收
- 合同变更 → PM 发起变更审批 → CEO 审批 → 合同金额更新

### 2.2 状态机转换测试 —— 未覆盖

每个业务实体有明确的状态，测试中只验证了初始状态，未测试任何状态转换。

#### 表单审批状态
- PENDING → APPROVED：审批通过后数据是否更新（考勤/加班记录生效）
- PENDING → REJECTED：被驳回后员工是否收到通知
- REJECTED → PENDING：重新提交后是否从审批链第一节点重新开始
- 已审批表单不可再编辑：前端是否禁用修改

#### 薪资周期状态
- WINDOW_OPEN → WINDOW_CLOSED：窗口关闭后员工无法新增提交
- SETTLED → LOCKED：结算后不可再修改
- LOCKED → UNLOCKED（CEO 解锁）→ 重新结算：数据是否正确重算

#### 员工账号状态
- ACTIVE → SUSPENDED：被停用账号登录应被拒绝
- SUSPENDED → ACTIVE：恢复后可正常登录
- 有未处理审批的员工被删除：孤儿审批如何处理

#### 项目状态
- 进行中 → 已完成 → 售后中 → 已完成：再次完成流程是否正常

### 2.3 CRUD 完整性 —— 只测了 Read

当前测试只验证列表页能否加载，未测试任何创建、编辑、删除操作。

- 员工管理：创建员工（必填字段完整提交）、修改姓名/部门、停用账号
- 岗位管理：创建岗位 + 薪级、修改薪级金额、删除岗位（有员工时是否阻止）
- 补贴配置：创建补贴类型、配置员工补贴覆盖
- 请假类型：HR 创建新假种、修改配额、禁用内置假种
- 项目：创建项目、添加里程碑、修改进度、完成里程碑
- 角色管理：自定义角色 CRUD、权限分配

### 2.4 文件上传 —— 未覆盖

- 报销申请发票附件：上传有效 JPG/PDF → 提交成功；上传超过限制数量 → 校验拦截；不上传 → 校验拦截
- 工伤申报附件：必填验证
- 通讯录 CSV 导入：有效 CSV → 预览正确；列名错误 → 报错提示；空文件 → 报错
- 施工日志附件上传
- 工程合同附件上传

### 2.5 数字计算准确性 —— 未覆盖

- 报销明细多行时，总金额是否等于各行之和（自动计算字段）
- 请假按小时扣款：请假2天 × 日薪基准 × 扣款比例 = 应扣金额
- 加班补贴：加班小时数 × 时薪 × 倍率（1.5×/2×/3×）
- 工资单合计：底薪 + 各项补贴 + 加班费 - 请假扣款 - 社保 = 实发工资

### 2.6 日期与时间边界 —— 部分缺失

- 请假结束日期早于开始日期 → 应校验拒绝
- 请假时长跨月（跨两个薪资周期）→ 扣款计入哪个周期
- 加班申报填写过去日期（窗口期外）→ 应拒绝
- 报销单出差日期早于入职日期 → 应校验
- 薪资周期日期重叠 → 创建第二个周期时应检查冲突

### 2.7 并发与重复提交 —— 未覆盖

- 双击提交按钮 → 是否产生两条记录
- 两个财务同时结算同一周期 → 结果是否一致
- 审批同一表单同时操作 → 是否有乐观锁或竞态保护

### 2.8 其他安全场景 —— 部分缺失

- 已登录用户 Token 过期后操作 → 应提示重新登录（当前未测试 Token 生命周期）
- 多设备同时登录同一账号 → 是否允许，后登录是否踢出前者
- 越权直接调用后端 API（跳过前端，使用 curl + 其他角色 Token）→ 后端是否独立拦截（BUG-E02 表明后端有控制，但前端守卫有漏洞）

## 三、补充测试优先级建议

#### 立即补测（影响演示/验收）
- 审批链完整流程：提交 → 审批 → 状态变化验证
- 薪资周期主链：创建 → 开窗 → 结算 → 签收
- 报销申请带文件上传的完整提交

#### 核心功能稳定后补测
- 全部 CRUD 正向路径（创建有效数据 → 修改 → 软删除）
- 状态机边界（驳回后重提交、解锁重算）
- 日期边界（开始>结束、窗口期外提交）

#### 质量加固阶段
- 数字计算精度验证
- 并发提交防重复
- Token 过期处理
- 后端越权 API 直调（独立于前端守卫的安全测试）
